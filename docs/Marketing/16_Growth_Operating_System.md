## 1. Growth OSの思想

### 1.1 4要素モデル

```
Growth OS = 人（Team） × プロセス（Process） × データ（Data） × リズム（Cadence）
```

| 要素 | 定義 | 壊れているときの症状 |
|---|---|---|
| 人 | 役割・権限・スキルの配置（§2） | 「誰が決めるのか分からない」「全部CEO待ち」 |
| プロセス | 仮説→実験→判定→展開の標準手順（15_Experiment_Backlog） | 施策が思いつきで走り、学びが残らない |
| データ | 単一の正となる指標定義と自動ダッシュボード（14_KPI_Dashboard） | 会議が「数字の確認」で終わり、意思決定に至らない |
| リズム | 日次〜年次の意思決定サイクル（§3〜§6） | 問題発見から対応まで数週間かかる |

### 1.2 OSの成功指標（Growth OS自体のKPI）

OSの品質も判定可能な数値で管理する。

| OS指標 | 定義 | 目標 |
|---|---|---|
| 実験スループット | 月間の判定完了実験数 | 0-6ヶ月: 月4本 → 12ヶ月: 月8本 → 24ヶ月以降: 月16本 |
| 実験勝率 | Win ÷ 判定完了数 | 20〜40%（40%超は挑戦不足、20%未満は仮説品質不足のシグナル） |
| 検知→対応リードタイム | アラート発火から一次対応までの時間 | 日次指標: 24時間以内 100% |
| 学習の記録率 | 終了後7日以内に結果記録が完成した実験比率 | 100% |
| 会議の意思決定率 | 定例会議のうち「決定事項が1件以上ある」回の比率 | 80%以上（下回る会議は棚卸し対象、§7） |

## 2. チーム体制

### 2.1 フェーズ別組織図

**Phase 1（0〜6ヶ月・3人体制）** — 全員がフルスタックグロース。

```
事業責任者（PM兼務）
├── グロース担当（実験・計測・CRM・広告）
└── コンテンツ担当（SEO記事・SNS・結果カードの言葉）
※ 開発はプロダクトエンジニア（既存開発チーム）と兼務協業
```

**Phase 2（7〜18ヶ月・10〜15人）** — 機能分化の開始。

```
事業責任者
├── Growth（獲得・実験）2〜3人
├── Product Growth（オンボ・継続・課金導線）2〜3人
├── Content（SEO/AEO・ショート動画）2〜3人
├── CRM（LINE/メール/プッシュ）1〜2人
└── Data（計測基盤・分析）1〜2人
```

**Phase 3（19ヶ月〜・50億円期に向けた6チーム体制）**

```
CGO（Chief Growth Officer）
├── Growthチーム: 獲得ループ運用・実験プログラム・チャネルポートフォリオ
├── Productチーム（グロース領域）: オンボーディング・習慣化・バイラル機能
├── Brandチーム: ブランド戦略・PR・UGC・健全性レビュー
├── Dataチーム: 計測基盤・実験プラットフォーム・LTVモデル
├── CRMチーム: ライフサイクルコミュニケーション・休眠復帰・解約防止
└── Contentチーム: SEO/AEO/LLMO資産・エディトリアル・ショート動画
```

### 2.2 チーム別ミッションと保有KPI（Phase 3体制）

| チーム | ミッション（1文） | 保有KPI（14_KPI準拠） | 主担当ループ（Spine §6） |
|---|---|---|---|
| Growth | 4ループの点火と獲得効率の最大化 | 新規登録数、CAC、K-factor、紹介経由比率、実験スループット | 全ループの統括 |
| Product（グロース領域） | 初回で共鳴させ、習慣にする | Aha率、D1/D7/D30、DAU/MAU、WRS | リテンションループ |
| Brand | 「脅さない占い」の想起1位と健全性 | NPS、指名検索数、UGC数/MAU、ガードレール指標 | UGCループ |
| Data | 全員が同じ数字で議論できる状態の維持 | 計測記録率、アラート誤検知率、LTVモデル精度 | （基盤） |
| CRM | 適切な人に適切な瞬間だけ声をかける | 通知経由WRS、オプトアウト率、休眠率、復帰率、解約率 | リテンションループ |
| Content | 悩みの検索・AI検索の入口を独占する | オーガニック流入、AI検索言及率、記事→登録CVR | コンテンツループ |

### 2.3 RACI（主要活動別）

R=実行 A=説明責任 C=協議 I=報告

| 活動 | Growth | Product | Brand | Data | CRM | Content | 事業責任者 |
|---|---|---|---|---|---|---|---|
| NSM/KPI定義変更 | R | C | C | R | C | C | A |
| 実験の優先順位確定 | R/A | C | C | C | C | C | I |
| 価格・プラン変更 | R | C | C | C | C | I | A |
| 通知・CRM配信設計 | C | C | C | I | R/A | C | I |
| ブランド倫理判定（§1.4除外リスト該当性） | C | C | R/A | I | C | C | 拒否権 |
| 大型機能のGo/No-Go | C | R | C | C | I | I | A |
| チャネル予算配分 | R/A | I | C | C | I | C | 四半期承認 |
| インシデント一次対応 | 類型による（§8） | | | | | | A |

### 2.4 意思決定権限 — 何をデータで決め、何を人間が決めるか

| 決定の種類 | 決め方 | 根拠 |
|---|---|---|
| 実験のWin/Lose判定 | **データが決める**（事前基準に機械的に照合。人の感想で覆さない） | 15_Experiment §2 |
| 実験の優先順位 | **データが決める**（RICEスコア順。逸脱は理由の記録が必要） | 15_Experiment §4 |
| アラート対応の要否 | **データが決める**（14_KPIの閾値超過=自動的に対応義務発生） | 14_KPI §3 |
| ブランド倫理・トーン | **人間が決める**（Brandが判定、事業責任者が拒否権。数値が良くても倫理除外リストに触れれば中止） | Spine §9 |
| 戦略スパインの改訂 | **人間が決める**（四半期レビューで事業責任者が判断、データは材料） | §6 |
| 新市場・大型投資 | **人間が決める**（データで確度を測り、賭けの判断は経営が負う） | 17_Roadmap ゲート |
| 出力（鑑定文）の品質基準 | **人間が決める**（満足度データは参考、最終は品質ガイドラインへの適合をレビュー） | — |

原則: **「可逆で頻繁な決定」はデータと現場に委譲し、「不可逆で稀な決定」は人間が集まって決める。** 会議は後者のためにだけ存在する。

---

### 2.5 意思決定の記録（Decision Log）

「不可逆で稀な決定」（§2.4）はすべてDecision Log（Notion正本）に記録する。記録なき決定は再燃する。

```markdown
# DEC-YYYY-NNN: 決定タイトル
- 日付 / 決定者（A） / 協議者（C）
- 決定内容: 1〜3行
- 選択肢と棄却理由: 検討した代替案と、選ばなかった理由
- 根拠データ: ダッシュボード/実験IDへのリンク
- 再検討条件: どの数値がどうなったらこの決定を見直すか（必須）
```

## 3. 運用リズム: 毎日（Daily）

### 3.1 毎朝見る指標（9:15までに担当が確認）

| 指標 | 確認内容 | 対応が必要になる閾値（14_KPI準拠） |
|---|---|---|
| DAU | 前週同曜日比 | -15%で当日中に原因特定着手 |
| WRS（7日移動累計） | トレンド+アンチメトリクス警告 | 週次-8%、または偽装警告フラグ点灯 |
| 新規登録数 | 前週同曜日比・チャネル内訳 | -20%でチャネル別に切り分け |
| 初課金数・MRR | 前週同曜日比 | -20%、または決済エラー率1%超 |
| 障害・炎上兆候 | エラー率、SNS言及の急増（リスニングツールの異常通知）、ストアレビュー急落、CS急増 | エラー率0.5%超 / ネガ言及が平常の3倍 / 1時間あたりCS平常の5倍 → §8インシデント発動 |

### 3.2 日次スタンドアップ（15分・9:30固定）

| 時間 | アジェンダ |
|---|---|
| 0〜5分 | ②グロースダッシュボード確認: 閾値超過アラートの有無 → あれば担当と対応期限を即決 |
| 5〜10分 | Running実験のヘルスチェック（サンプル蓄積ペース・ガードレール異常・早期停止条件該当の有無のみ。**途中の勝ち負けは見ない**＝覗き見によるバイアス防止） |
| 10〜15分 | 今日のブロッカー共有（各自30秒。議論が必要なら別途15分枠を即設定し、スタンドアップは延長しない） |

- 議事は3行以内でSlack `#growth-daily` に投稿。「アラートなし・実験正常」の日は1行でよい

## 4. 運用リズム: 毎週（Weekly）

### 4.1 グロース定例（60分・月曜14:00）

| 時間 | アジェンダ | アウトプット |
|---|---|---|
| 0〜5分 | 先週確定した実験の学び共有（15_Experiment §7.3） | アーカイブ更新確認 |
| 5〜20分 | **先週の実験結果判定**: 判定期限が来た実験を事前基準に照合してWin/Lose/Inconclusive確定 | 判定記録・展開/ロールバック指示 |
| 20〜35分 | **今週の実験着手**: バックログRICE上位から着手枠（レイヤー空き）に割り当て。起票された新実験のスコアレビュー | 今週のRunning一覧 |
| 35〜50分 | **ファネル異常確認**: 週次でCVRファネル4段（14_KPI §3.6）+継続+収益+バイラルの各面を確認。閾値超過の深掘りと担当割当 | 異常対応タスク |
| 50〜60分 | 週次目標の確認（NSM進捗、OKRのKR進捗%） | 週次レポート承認 |

### 4.2 コンテンツ振り返り（30分・木曜）

- 先週公開コンテンツの流入・登録CVR・AI検索言及の確認
- SEO/AEO/LLMOのポジション変動（13_SEO_AEO_LLMO準拠の監視セット）
- 今週の制作パイプライン確認（公開予定と担当）

### 4.3 週次レポートフォーマット（金曜EODにSlack+Notion、A4一枚相当を厳守）

```markdown
# Weekly Growth Report — YYYY-WW
1. NSM: WRS xx,xxx（前週比 +x.x%）/ WRS/WAU xx%（アンチメトリクス警告: 有/無）
2. 5指標サマリ: 月商計画比 / MAU / D30 / LTV/CAC / 紹介経由比率 — 各: 実績・計画比・矢印
3. 実験: 判定x本（Win x / Lose x / Inc x）・着手x本・主要な学び1行×判定数
4. 異常と対応: 閾値超過x件 → 各対応状況1行
5. 来週の焦点: 3行以内
```

## 5. 運用リズム: 毎月（Monthly）

月次レビューは月初第3営業日に半日ブロックでまとめて実施する。

| レビュー | 内容 | 判定すること |
|---|---|---|
| コホートレビュー | 14_KPI §5の標準3問（直近コホートは6ヶ月前より良いか / LTV/CAC<3のチャネルはどれか / 最悪の解約セグメントはどこか） | プロダクト改善が効いているかの月次判定。2ヶ月連続でコホート非改善なら実験ポートフォリオを継続系に再配分 |
| CAC/LTVレビュー | チャネル別CAC・LTV12M・回収月数の更新。有料広告経由比率のSpine上限20%チェック | チャネル別の増額/維持/減額（増額はLTV12M/CAC≥3のチャネルのみ許可） |
| CRMシナリオ改善 | シナリオ別成果（⑥CRM面）・オプトアウト率・通知経由セッションの質 | 最下位シナリオ1本の改廃。配信総量の見直し |
| 価格・プラン構成レビュー | プランミックス・ARPPU（目標¥3,000）・プラン別解約率・従量課金購入率 | 価格実験（REV系）の起票要否。※実際の価格改定は四半期判断 |
| ブランド健全性レビュー | ガードレール指標全件（14_KPI §8）・NPSコメント・SNSセンチメント・CS品質 | ガードレール悪化があれば該当施策の特定と是正を**グロース目標より優先して**決定 |

## 6. 運用リズム: 毎四半期（Quarterly）/ 毎年（Yearly）

### 6.1 四半期（1日オフサイト）

| セッション | 内容 |
|---|---|
| OKR振り返り | KR達成率の確定（0.0〜1.0採点）。達成率と「学び」を分離して評価（挑戦的KRの0.7は良い成績） |
| OKR設定 | §6.3のカスケードで次QのO/KRを確定 |
| 戦略スパイン改訂判断 | Spine前提（NSM定義・目標値・4ループ・ARPPU前提）に反証データが出ていないかを審査。改訂はSpine→影響ドキュメントの順（Spine冒頭規約） |
| 大型実験のGo/No-Go | Effort 8人週超の「賭け」案件（Wrapped、相性占い等）の判定。判定材料: 先行小実験の結果+RICE+ロードマップ整合 |
| チャネルポートフォリオ再配分 | チャネル別コホートLTV/CACに基づく予算・人員の再配分。サンセット基準該当チャネルの縮小決定（§7） |
| ペルソナ更新 | 新規コホートの実態（年齢・流入文脈・利用ジョブ）と03_Personaの乖離確認、必要なら改訂 |
| プロセスレトロ（半期ごと） | §9参照 |

### 6.2 年次

| 項目 | 内容 |
|---|---|
| 事業計画 | 17_Roadmapの年次改訂。3シナリオ（保守/基本/強気）の見直しと投資計画 |
| ブランド調査 | 認知率・想起順位・ブランドイメージの定量調査（n≥1,000）。指名検索数と突合 |
| Wrapped型イベント | 年末「あなたの1年の星の記録」の企画・実行（最大のバイラル資産、REF-10の年次本番） |
| 市場再分析 | 市場規模・競合・技術（LLM性能/コスト）・規制の再スキャンと01/02ドキュメント改訂 |

### 6.3 OKRの立て方（NSMからのカスケード例）

- O（Objective）は定性・鼓舞的、KR（Key Result）は判定可能な数値。KRは3個以内
- 会社O → チームO → 個人KRへ**指標の因果で**接続する（分担表の切り貼りにしない）

```
会社O: 「開くたびに心が軽くなるサービス」として習慣の一部になる
├─ KR1: WRS/WAU 28% → 35%
├─ KR2: D30継続率 12% → 15%
└─ KR3: 月商 ¥1.2億 → ¥2.5億（12ヶ月目標到達）

  Growth チームO: 共鳴が自然に人を連れてくる状態を作る
  ├─ KR1: 紹介経由新規比率 18% → 25%
  ├─ KR2: シェア完了率/保存UU 4% → 8%
  └─ KR3: 実験スループット 月8本・記録率100%

    個人（グロース担当A）KR例:
    ├─ KR1: 相性占い招待フロー（REF-02）をQ内にWin判定まで到達させる
    ├─ KR2: K-factor 0.18 → 0.25
    └─ KR3: 招待→登録CVR 25% → 32%
```

- **アンチパターン禁止**: KRに「◯◯をリリースする」（アウトプット）だけを置かない。リリースは手段、KRは指標変化で書く（例外: 計測基盤等のイネーブラは四半期1個まで許可）

## 7. 「やめる」仕組み

成長組織の失敗の大半は「始めすぎ・やめなさすぎ」で起きる。やめる判断を制度化する。

| 対象 | サンセット基準（判定文） | プロセス |
|---|---|---|
| チャネル | **3ヶ月連続で目標新規獲得数の50%未満**、またはLTV12M/CAC<1.5が2四半期継続 → 縮小検討。さらに1四半期改善なしで停止 | 四半期ポートフォリオ再配分で決定。停止チャネルは負け仮説リストに敗因を記録 |
| 機能 | **3ヶ月連続で利用率（利用UU/MAU）が目標の50%未満** かつ WRS寄与が計測不能に小さい → 縮小検討（導線降格→非表示→削除の3段階） | 月次コホートレビューで検知、四半期で決定 |
| 実験 | 早期停止3条件（15_Experiment §1.2）該当で即時打ち切り。28日で必要サンプル未達ならInconclusive確定して終了（ダラダラ延長禁止） | 週次定例 |
| CRMシナリオ | 目標CVRの50%未満が2ヶ月連続 → 改廃（月次CRMレビュー） | 月次 |
| 会議 | 意思決定率80%未満が1四半期継続 → 廃止・統合・短縮のいずれかを必ず選ぶ | 四半期オフサイトで会議棚卸し |
| ダッシュボード指標 | 3ヶ月間どの意思決定にも使われなかった指標は削除候補（14_KPI §9） | 四半期 |
| 定常業務 | 半期プロセスレトロ（§9）で「この業務をやめたら何が壊れるか」を全定常業務に問う。答えられない業務は1ヶ月停止テスト | 半期 |

## 8. インシデント運用（4類型プレイブック）

全類型共通: 検知（§3.1の閾値）→ 15分以内に `#incident` にインシデント宣言 → 類型別プレイブック起動 → 収束後5営業日以内にポストモーテム（非難なし文化・再発防止アクション必須）。

### 8.1 類型① 炎上（SNS上の批判拡散）

| 項目 | 内容 |
|---|---|
| 検知 | ネガ言及が平常の3倍/時、またはメディア取材連絡 |
| 一次対応（1時間以内） | Brandがオーナー。事実確認（何が・どこまで真実か）を最優先。**確認前の反論・削除対応は禁止** |
| 対応原則 | 誤りがあれば24時間以内に具体的な謝罪+是正策を公式発信。誤解には冷静な事実提示のみ。個別ユーザーとの応酬禁止 |
| 事前準備 | 想定問答集（占いの科学性・依存性・データ扱い・料金への批判）を四半期更新 |

### 8.2 類型② 誤鑑定・不適切出力（AIが有害/不適切な内容を出力）

| 項目 | 内容 |
|---|---|
| 検知 | ユーザー通報、出力監査サンプリング（週次100件）、不安系ワード監視（14_KPI §8） |
| 一次対応（30分以内） | Productがオーナー。該当出力パターンの特定→プロンプト/フィルタの即時修正またはロールバック。深刻例（自傷・医療・法律の断定等）は該当機能の一時停止も選択肢 |
| ユーザー対応 | 影響ユーザーへ個別連絡+無償対応。深刻例は相談窓口（公的窓口含む）の案内 |
| 恒久対応 | 出力禁止事項リストの更新、リグレッションテストへの追加 |

### 8.3 類型③ 依存批判（「依存させて課金させている」との指摘）

| 項目 | 内容 |
|---|---|
| 検知 | メディア/SNS/CSでの依存性への批判、ガードレール指標の悪化（14_KPI §8） |
| 一次対応 | Brand+Productが共同オーナー。まずデータで自己診断（依存過多率・深夜利用率・高額課金集中率の実値確認）。**批判が正しければ争わず直す** |
| 対応原則 | 健全利用機能（利用リズム提案・ソフトキャップ・「今日はおやすみ」ナッジ）の実装状況を公開できる状態を平時から維持する。これはSpine §9の通り最大の差別化資産 |
| エスカレーション | 実データで依存指標が閾値超過していた場合、是正をロードマップ最優先に割り込ませる（経営決定事項） |

### 8.4 類型④ 障害（システム停止・決済不能・データ漏洩）

| 項目 | 内容 |
|---|---|
| 検知 | エラー率0.5%超、決済失敗率1%超、監視アラート |
| 一次対応（15分以内） | 開発オンコールがオーナー。SEV判定（SEV1: 全停止/決済/漏洩、SEV2: 主要機能、SEV3: 部分） |
| ユーザー告知 | SEV1は30分以内にステータス告知（アプリ内+SNS）。復旧見込みを楽観で言わない |
| データ漏洩特則 | 相談内容はセンシティブ。漏洩疑い時点で法務+外部専門家+当局報告フローを即起動（個人情報保護委員会への報告義務基準を事前確認済みにしておく） |
| CRM停止判定 | 障害中の販促配信は全停止（障害中に売り込む姿はブランド毀損） |

### 9.1 ナレッジ更新義務（「更新されない文書は存在しない文書」）

| 資産 | 更新トリガー | 責任 |
|---|---|---|
| 実験アーカイブ | 実験終了+7日以内（記録率100%がOS指標） | 実験オーナー |
| プレイブック（インシデント・チャネル運用・CRM） | 使用のたびに「実際と違った箇所」を追記。四半期に棚卸し | 各オーナー |
| オンボーディング資料（新メンバー用） | 新メンバー参画のたびに本人が「分からなかった点」を改訂して次代に渡す（更新を入社タスクに含める） | 直近の新メンバー |
| KPI定義（14）・実験規約（15）・本書（16） | 改訂はPR+オーナー承認、Version更新 | 各Owner |
| 週次/月次レポートアーカイブ | 自動蓄積（Notion）。年次で「今年の10大学び」に編纂 | Growth |

### 9.2 半期プロセスレトロスペクティブ（Growth OS自体の改善）

半期に1回、半日でOS自体をレトロする。アジェンダ:

1. **OS指標レビュー**（§1.2の5指標）: スループット・勝率・リードタイム・記録率・会議の意思決定率
2. **Keep/Problem/Try**: リズム（会議体・レポート）、プロセス（実験規約・RACI）、データ（ダッシュボード・アラート精度=誤検知率）の3領域で実施
3. **やめるものの決定**: §7の全対象の棚卸し実行
4. **本ドキュメントの改訂**: レトロの結論はその場で本書のPRにする（「議事録止まり」を禁止）

- アラート閾値の校正もここで行う: 過去半年の発火履歴を見て、誤検知率>50%の閾値は緩め、一度も発火せず問題を見逃した閾値は厳しくする
- レトロの結論が3回連続で同じProblemを挙げていたら、それはプロセスでなく構造（人員・権限）の問題として経営判断にエスカレーションする
