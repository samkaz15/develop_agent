## 1. 実験運用の原則

### 1.1 実験サイクル（例外なくこの順で回す）

```
仮説（Because we saw X, we believe Y） 
→ 最小実験（判定に必要な最小の変更・最小の対象範囲）
→ 計測（事前登録した主要指標+ガードレール指標のみで判定）
→ 判定（Win / Lose / Inconclusive の3値。事前基準に照らして機械的に）
→ 展開（Win: 100%展開+ナレッジ記録） / 棄却（Lose: ロールバック+ナレッジ記録）
```

- **事前登録主義**: 主要指標・判定基準・期間は実験開始前にテンプレート（§3）へ記入し、開始後の変更を禁止する。「後から効いていた指標を探す」ことをp-hackingとして禁止
- **1実験1仮説**: 複数の変更を混ぜた実験は、勝っても負けても学びが特定できないため原則禁止（例外: リデザイン等の「束の実験」は事前にそう宣言する）

### 1.2 サンプルサイズと検定の実務基準

| 項目 | 基準 |
|---|---|
| 統計基準 | 両側検定、有意水準 α=0.05、検出力 80% |
| 最低検出効果（MDE） | CVR系: 相対 +10%（例: 転換率6%→6.6%）。継続率系: 絶対 +2pt。これより小さい効果を狙う実験は原則着手しない（検出に必要なサンプルが大きすぎ、投資対効果が合わない） |
| 必要サンプルの目安 | ベースCVR 6%・相対MDE 10%の場合、1群あたり約26,000UU。ベース40%・絶対2ptなら1群あたり約9,400UU（実験設計時に必ず計算機で算出しテンプレートに記入） |
| 実験期間の決め方 | `期間 = 必要サンプル ÷ 対象への日次流入`。ただし**最低7日間（曜日効果を1周させる）、最大28日間**。28日で必要サンプルに届かない実験は、対象を広げるかMDEを引き上げて再設計 |
| 早期停止 | 原則禁止。例外は (a)ガードレール指標がアラート閾値超過、(b)重大バグ、(c)ブランド毀損の兆候、の3つのみ。この場合は Inconclusive として記録 |
| 継続率実験の特則 | D7判定は開始後「期間+7日」、D30判定は「期間+30日」まで判定を待つ。待てない場合はD7を代理指標にし、D30で追認する二段判定を事前宣言 |
| サンプル不足期の運用 | MAU10万人未満のフェーズでは厳密なA/Bが成立しない実験が多い。その場合は (a)前後比較+ホールドアウト10%、(b)定性テスト（ユーザーテスト5名で致命的問題を潰す）に切り替え、「準実験」として確信度Cを1段下げて記録 |

### 1.3 同時実験の干渉管理

- **レイヤー制**: ①獲得（LP/広告）②オンボーディング ③コア体験（鑑定・通知）④課金（ペイウォール/価格）の4レイヤーを定義。異なるレイヤーの実験は同時実行可、同一レイヤー内は原則1本ずつ
- 同一レイヤーで並行する場合は、ユーザーを相互排他バケットに分割（実験プラットフォームのmutual exclusion group）
- 全イベントに `experiment_ids[]` を付与（14_KPI §6.1）し、事後に交互作用を検査できる状態を保つ
- 価格系実験（P-系）は同時に1本のみ。かつ既存課金者を対象に含めない（値上げの遡及適用禁止）

### 1.4 実験しないもの（ブランド倫理の除外リスト）

以下はどれだけ数値貢献が見込めても**実験自体を禁止**する（Spine §9）。

1. 恐怖訴求・不幸の断定を用いたコピー/通知（「このままだと不幸になります」等）
2. 依存を誘発する設計（無限続行の煽り、深夜利用を促す通知、損失回避を突く連続ログイン脅迫）
3. 解約・退会を意図的に困難にするダークパターン
4. 偽の社会的証明（架空レビュー、水増しカウンター）
5. 未成年への課金誘導強化
6. 相談内容（センシティブデータ）を利用したターゲティング広告
7. ガードレール指標（14_KPI §8）を悪化させることが事前に自明な施策

### 1.5 サンプルサイズ早見表（α=0.05両側・検出力80%・1群あたり必要UU）

設計時はツールで正確に計算するが、起票時の実現性判断には本表を使う。

| ベース値 | 相対+10% | 相対+15% | 相対+20% | 相対+30% | 相対+50% |
|---|---|---|---|---|---|
| 2%（例: シェア率） | 78,000 | 35,000 | 20,000 | 9,000 | 3,400 |
| 6%（例: 転換率） | 26,000 | 11,600 | 6,600 | 3,000 | 1,100 |
| 12%（例: 訪問→登録） | 12,000 | 5,400 | 3,100 | 1,400 | 530 |
| 40%（例: D1・Aha率） | 1,500 | 680 | 390 | 180 | 70 |
| 60%（例: 登録完了率） | 660 | 300 | 170 | 80 | 30 |

**含意**: 低ベース率指標（シェア・転換）の小さな改善は初期フェーズでは検出不能。低ベース指標は大きな変更（相対+30%以上を狙う大胆な実験）でのみ挑む。高ベース指標（D1・Aha率）は小さな改善でも検定が回るため、細かい磨き込みはこちらに寄せる。

### 1.6 実験できないもの（能力上の限界と代替手段）

倫理除外（§1.4）とは別に、統計的に実験が成立しない領域の扱いを定める。

| 領域 | 理由 | 代替検証手段 |
|---|---|---|
| ブランド認知施策（PR・マス） | 対照群が作れない | 地域split（可能な場合）、指名検索数の前後時系列+因果推論（CausalImpact） |
| SEO/AEO施策 | 検索エンジン側の変動と分離不能 | 記事群単位のstaggered rollout、対照キーワード群との差分 |
| 大型リデザイン | 変更の束で要素分解不能 | 事前に「束」と宣言し、主要指標のみで判定。要素の学びは主張しない |
| 年1回イベント（Wrapped） | 反復不能 | 前年比+コホート内利用者/非利用者の傾向スコアマッチング（参考値扱い） |

## 2. 判定基準の標準（全実験共通）

| 判定 | 条件 |
|---|---|
| **Win** | 主要指標が事前MDE以上に有意に改善 かつ 全ガードレール指標が非悪化（悪化=事前定義の許容幅超過） |
| **Lose** | 主要指標が有意に悪化、またはガードレール指標が許容幅を超えて悪化 |
| **Inconclusive** | 上記以外（有意差なし・サンプル不足・早期停止）。同一仮説の再実験は設計を変えた場合のみ1回まで許可 |

**標準ガードレール指標セット**（全実験に自動適用、実験ごとに追加可）: D1継続率 / 鑑定満足度 / 解約率（課金系のみ） / 通知オプトアウト率（CRM系のみ） / 依存過多ユーザー率（14_KPI §8）。

```markdown
## EXP-XXX: 実験名
| 項目 | 内容 |
|---|---|
| 仮説 | （観察事実）から、（変更）により（対象）の（指標）が（幅）改善すると考える |
| 対象 | セグメント・プラットフォーム・流入割合（例: 新規登録者の50%、Webのみ） |
| 変更 | Control と Variant の差分を1文で。スクリーンショット/コピー全文を添付 |
| 主要指標 | 1つだけ（14_KPI定義準拠）。MDEと必要サンプル数を明記 |
| ガードレール指標 | 標準セット+追加分。各許容幅 |
| 期間 | 開始日〜終了日（最低7日、最大28日）。継続率判定の待機期間 |
| 判定基準 | Win/Lose/Inconclusiveの数値条件（§2準拠で具体化） |
| レイヤー | 獲得/オンボ/コア/課金 |
| 結果記録 | 判定・実測値・信頼区間・学び（Whyの考察）・次のアクション ※終了後7日以内に記入 |
```

## 4. ICE/RICE運用定義

優先度スコアは **RICE = (Reach × Impact × Confidence) ÷ Effort** を正本とする。ICEはバックログ起票時の簡易仕分け（各1〜5の直感点、合計10以上でRICE精査に進む）にのみ使う。

### 4.1 採点基準（明文化）

| 要素 | 定義 | 採点基準 |
|---|---|---|
| **Reach** | 四半期内に変更へ接触するUU数（千人単位） | 実データから推計。例: 新規オンボ実験でQ内新規9万人×対象50% → R=45 |
| **Impact** | 主要指標への期待効果 | 3=巨大（主要KPI相対+20%以上見込み）/ 2=大（+10-20%）/ 1=中（+5-10%）/ 0.5=小（+2-5%）/ 0.25=微小 |
| **Confidence** | 仮説の裏付け強度 | 100%=自社データ+他社事例の両方 / 80%=自社データあり / 50%=定性シグナルのみ / 20%=直感 |
| **Effort** | 人週（設計+実装+分析の合計） | 0.5刻み。1未満は1に切り上げ |

- スコアの採点はオーナー起票→週次グロース定例で第三者1名がレビューし確定（自己申告の Impact インフレを防ぐ）
- スコア順を機械的に守る。順位を飛ばして着手する場合は定例で理由を記録（戦略的依存関係・季節性のみ許可）

### 5.9 バックログ運用ルール

- 起票は誰でも可（テンプレート§3の仮説・主要指標まで記入必須）。週次定例でICE仕分け→RICE確定
- 四半期ごとに全RICEスコアを再計算（Reachは成長で変わる）。Icebox行きの基準: RICE<5 かつ 戦略依存なし
- 同時Running上限: 4本（レイヤーごと1本、§1.3）。Design中は上限なし

## 9. 06_Product_Improvement.md とのID対応方針
