## 1. ペルソナ設計の方針

### 1.1 三原則

| # | 原則 | 判定文（これを満たさなければ違反） |
|---|---|---|
| 1 | **JTBDが核、デモグラは服** | 全施策はペルソナIDではなくジョブID（J-F/J-E/J-S）まで指定して設計されている |
| 2 | **データで更新される生きたペルソナ** | 各ペルソナに行動データ上の判定条件（セグメント定義SQL相当）が存在し、四半期レビューで実測値（LTV・継続率・シェア率）と本書の想定値の乖離が±30%以内に保たれている。乖離が超えたらペルソナ定義を改訂する |
| 3 | **獲得しない層を先に決める** | アンチペルソナ（§6）に該当するユーザーへの獲得投資はゼロ。該当検知時は保護・制限・案内のいずれかのフローに入る |

ペルソナは登録時アンケート（3問: 相談テーマ / 年代 / きっかけ）+ 行動ログで推定し、`persona_segment` としてユーザー属性に付与する。

## 2. JTBD分析 — 「人は占いを買うのではなく、ジョブを片づけるために占いを雇う」

Spine §1 の通り、本サービスのジョブは「当てること」ではなく **「不安が言語化され、心が軽くなり、明日の一歩が決まる」** こと。ジョブは機能的（F）・感情的（E）・社会的（S）の3層で定義する。

### 2.1 ジョブステートメント一覧（12個）

書式:「【状況】のとき、【動機】したい、そうすれば【期待成果】」

#### 機能的ジョブ（Functional）

| ID | ジョブステートメント | 主ペルソナ | 対応する提供価値 |
|---|---|---|---|
| J-F1 | **選択肢が2つ以上あって決めきれない**とき、**自分では気づけない判断材料と背中の一押しが**欲しい、そうすれば**先延ばしをやめて行動に移せる** | P2, P5, P8 | 対話鑑定→「明日やること」1つの提示 |
| J-F2 | **頭の中で悩みがぐるぐるして整理できない**とき、**問いを立てて言語化する手伝いが**欲しい、そうすれば**何に悩んでいるのか自体が明確になる** | P1, P2, P7 | 対話での深掘り・悩みの構造化サマリ |

### 2.2 ジョブ優先順位の結論

- **雇用頻度×切実さ**が最大なのは **J-E1（深夜の不安）と J-E5（誰にも言えない悩み）**。ここがAIの構造的優位（24時間・非審判・従量コストほぼゼロ）と重なるため、**プロダクトの中核体験はJ-E1/J-E5に最適化する**
- **バイラルの燃料**は J-S1/J-S3。獲得コンテンツはここから設計する（→ [08_Viral_Strategy.md](./08_Viral_Strategy.md)）
- **高単価化の根拠**は J-F1/J-F4（意思決定支援）。Premium（¥9,800）はこのジョブの完遂度で正当化する（→ [10_Pricing.md](./10_Pricing.md)）

## 5. 行動経済学・心理学の適用表

Spine §9 の倫理（恐怖訴求・依存誘発・断定的不幸予言の禁止）の範囲内でのみ適用する。**各原理に「禁止される使い方」を併記する**。

## 6. アンチペルソナ — 獲得してはいけない/してはいけない扱いの層

### 6.1 AP1: 依存傾向が強く自傷リスクがある層 — 「保護対象」

**位置づけ**: 排除ではなく**保護設計の対象**。ここでの誠実さがブランド資産（Spine §9）そのものになる。

| 項目 | 内容 |
|---|---|
| 検知条件（いずれか） | (1) 1日の利用が累計3時間超が7日continuing (2) 同一テーマの反復相談が1日10セッション超 (3) 自傷・希死念慮に関するキーワード検出 (4) 深夜帯利用が30日連続 |
| 対応設計 | (3)検出時は鑑定を中断し、共感的メッセージとともに公的相談窓口（いのちの電話等）へ即時案内。(1)(2)(4)は「今日はもう十分向き合いました」の健全化ナッジ、利用時間サマリの可視化、クールダウン提案 |
| 収益上の扱い | この層への課金促進通知・アップセルは**システム的に停止**する。判定文: 「検知条件該当ユーザーへのプロモ配信数 = 0」 |
| 禁止事項 | 依存を収益源とすること。断定的な不幸予言。「もっと占わないと危ない」型の訴求 |

### 6.2 AP2: クレーマー層（攻撃的・要求過剰） — 「距離を置く」

| 項目 | 内容 |
|---|---|
| 特徴 | 「当たらなかった」ことへの返金要求の反復、サポートへの攻撃的言動、規約外の要求 |
| 検知条件 | 返金要求2回以上 かつ NPS 0-2 かつ サポート起票月3件以上 |
| 対応設計 | 期待値の事前明示（「本サービスは未来を断定しません」を登録時に明記）で発生自体を予防。発生時はテンプレ化された誠実な一次対応→改善しない場合は利用規約に基づく利用制限。**個別対応に事業責任者の時間を使わない** |
| 獲得上の扱い | 「絶対当たる」系の誇大訴求クリエイティブはこの層を集めるため禁止（獲得段階で予防する） |

### 6.3 AP3: 無料乞食層（Free Rider） — 「コストを掛けず、資産化する」

| 項目 | 内容 |
|---|---|
| 特徴 | 複数アカウントで無料枠を回遊、課金意図が構造的にゼロ。ただし**シェア・UGC・WRSには貢献し得る** |
| 検知条件 | 登録90日超 かつ 課金画面到達0回 かつ 無料枠使い切り率90%超 |
| 対応設計 | (1) 複垢は端末・決済フィンガープリントで無料枠を統合 (2) 排除はしない——無料層はUGCループの燃料。ただし高コストな長文深掘り鑑定は有料側に置き、無料枠の限界原価を管理 (3) この層向けのCRM配信はシェア促進・紹介促進のみに限定（課金訴求は打たない） |
| 判定文 | 無料ユーザー1人あたり月間推論コスト ≤ 目標値（[10_Pricing.md](./10_Pricing.md)で定義）を維持し、超過時は無料枠仕様を見直す |

### 6.4 アンチペルソナ運用の共通原則

1. アンチペルソナの検知ロジックは [14_KPI_Dashboard.md](./14_KPI_Dashboard.md) の管理画面で週次モニタリングする
2. AP1保護対応の発動件数と対応完了率は、収益KPIと同格の「信頼KPI」として月次報告する
3. 広告・クリエイティブ審査時に「この表現はAP1/AP2を集めないか」をチェック項目に含める
