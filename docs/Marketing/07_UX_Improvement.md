## 1. UX原則7箇条

すべての画面・機能はリリース前に以下7箇条でレビューし、1つでも違反があれば出荷しない。

| # | 原則 | 判定文 |
|---|---|---|
| 1 | **3タップ以内で初回鑑定に到達する** | アプリ/LP到達から鑑定開始までのタップ数が3以下である |
| 2 | **初回セッション10分以内にAha Momentを作る** | 初回セッションで「個人的に言い当てられた」と感じる洞察が1つ以上、開始10分以内に提示される |
| 3 | **すべての画面はスクショされる前提でデザインする** | どの結果画面を切り取っても、①見た目が美しく ②サービス名が入り ③個人特定情報が写り込まない |
| 4 | **不安を煽る表現を出さない** | 「このままだと不幸になる」型の恐怖訴求・断定的不幸予言が全出力・全UXコピーに存在しない（06 #88の自動ガードで機械的に保証） |
| 5 | **待ち時間は演出に変える** | 2秒を超える待ちには必ず意味のある演出（カードシャッフル・星盤描画）があり、無言のスピナーが存在しない |
| 6 | **鑑定は必ず「希望+今日の一歩」で終わる** | すべての鑑定の最終ブロックが前向きな見通しと実行可能な行動1つで構成される（ピークエンドの法則） |
| 7 | **ユーザーの自己決定を奪わない** | 「決めるのはあなた」という態度が一貫し、依存を誘発するダークパターン（過剰な残数煽り・解約妨害）が存在しない |

## 2. 初回体験・オンボーディング

### 2.1 設計思想: 登録前に価値を先に出す

登録フォーム→価値、の順番を逆転させる（06 #99）。**生年月日を入れた瞬間にミニ鑑定が始まり、結果の途中で「続きを見るには登録」**が正しい順序。人は空のフォームには入力しないが、始まった物語は途中でやめられない。

### 2.2 入力ステップの上限

| ルール | 基準値 |
|---|---|
| 登録前に要求してよい入力 | 生年月日のみ（+任意で呼び名） |
| 初回セッションで要求してよい入力項目数 | 3項目以内（呼び名・生年月日・関心テーマ3択） |
| 1画面あたりの入力項目数 | 1項目（1画面1質問） |
| 出生時刻・出生地・恋愛状況等 | 初回では聞かない。プログレッシブプロファイリング（06 #101）で「もっと精密に視るために」と価値と交換で後日取得 |
| 入力のたびに返す即時フィードバック | 必須。生年月日→星座と一言、出生時刻→月星座（06 #105）。フォーム自体を鑑定体験の一部にする |

### 2.3 初回体験タイムライン（0秒〜10分の設計）

| 時刻 | 体験 | 設計意図 | 計測ポイント |
|---|---|---|---|
| 0〜5秒 | ファーストビュー。「今夜、何に迷っていますか」の一文と生年月日入力欄のみ。説明文・機能紹介なし | 迷いを持つ人への呼びかけで自分ごと化。チュートリアル排除（06 #109） | LP直帰率 |
| 5〜30秒 | 生年月日入力→即座に星座・今日の空模様を1行提示 | 最初の入力への即時報酬 | 入力開始率 |
| 30秒〜1分 | 関心テーマ3択（恋愛/仕事/自己理解） | 選択が初期パーソナライズデータになる（06 #104） | テーマ選択率 |
| 1〜2分 | カードシャッフル演出（3〜5秒）→ミニ鑑定の前半表示 | 待ち時間を儀式に変える（原則5） | 演出完了率 |
| 2〜4分 | ミニ鑑定本文。ここで最初のAha（性格・今の状況の言い当て）を必ず1つ入れる | 原則2の実装。Ahaは「悩みの言語化」で作る | 鑑定読了率 |
| 4〜5分 | 結果の核心の直前で「続きを見るには登録（LINEで10秒）」 | 価値実感後の登録要求。ソーシャルログイン優先（06 #103） | 登録転換率 |
| 5〜7分 | 登録完了→結果の後半+「今日の一歩」提示 | 原則6。初回でも必ず希望と行動で締める | 初回鑑定完了率 |
| 7〜9分 | 「明日の朝、今日の続きが視えます」の予告+通知許諾の依頼 | 通知許諾を「答え合わせのため」の文脈で（06 #106, #107） | 通知許諾率 |
| 9〜10分 | 結果カードの保存/シェア提案→ホームへ | 初回セッション内でWRS（保存/シェア）到達を狙う | 初回保存・シェア率 |

**判定基準**: 初回到達者のうち10分以内に「鑑定完了+保存/シェア/通知許諾のいずれか」に到達する割合を **Time to Aha 達成率** とし、目標50%以上（§11）。

## 3. 診断・鑑定体験

### 3.1 質問の受け止め方

鑑定はいきなり答えから始めない。**「オウム返し+共感→鑑定」**の2段構えを標準とする。

1. **受け止め**: 相談内容を要約して返す（「彼からの返信が減って、気持ちが離れたのか不安なんですね」）— 1〜2文、評価や助言を含めない
2. **共感**: 感情の正当化（「その状況で不安になるのは自然なことです」）— 1文
3. **鑑定へ**: 「カードに聞いてみましょう」で占いの世界へ移行

相談が曖昧な場合は深掘りの逆質問を最大2問（06 #10）。3問以上は尋問になる。

### 3.2 インタラクション演出

| 演出 | 仕様 | 意図 |
|---|---|---|
| タロット | ユーザーが自分でスワイプしてシャッフル→タップでカードを1枚ずつめくる | 「自分で引いた」という当事者性が結果の受容度を上げる |
| 星盤 | ホロスコープが線を描きながら3〜4秒で描画される | 計算している感=根拠の演出（06 #3と連動） |
| 演出時間の上限 | 初回5秒以内、2回目以降はスキップ可能 | 儀式感と効率の両立。毎回強制すると習慣利用の摩擦になる |

### 3.3 結果表示のリズム: 段階開示

結果は一気に出さない。**結論→（タップ/スクロール）→理由→（タップ）→今日の一歩**の3段階で開示する（06 #4の3部構成に対応）。

- 各ブロックは表示時にフェードイン。次ブロックへの誘い文（「なぜそう出たのか、視てみましょう」）を挟む
- 1ブロックはスマホ1画面（約200〜300字）に収める
- 全文を最初からスクロール可能にするのは既読ユーザーの再読画面のみ

### 3.4 ピークエンド設計

体験の記憶は「ピーク」と「最後」で決まる。

- **ピーク**: Aha（言い当て）またはカードめくりの瞬間に置く
- **エンド**: 必ず「希望+今日の一歩」で締める（原則6）。ネガティブなカードが出た場合も「注意」で終わらせず、「だからこそ今日できること」へ変換して締める。変換できない鑑定文は品質ガード（06 #22）で再生成

## 4. チャット体験

| 項目 | 基準 | 補足 |
|---|---|---|
| 初動応答 | 1秒以内に「タイピング中」表示を開始 | 無反応時間ゼロ。生成が遅い場合も受け止め（§3.1の1文）を先に返す |
| 応答の長さ | 1メッセージ3文以内を基本、長い鑑定は複数メッセージに分割して順次表示 | 長文の壁はチャットでは読まれない |
| 感情への相槌 | 感情語（つらい・不安・嬉しい）を検知したら、内容の前にまず相槌を1文 | 06 #5の感情検知と連動 |
| 会話メモリー | 同一セッション内は全文脈、セッション跨ぎは要約メモリーで「先週の件」を参照可能に | 06 #2。「覚えていてくれる」がチャット体験の核 |
| 深夜モード | 23時〜4時はトーンを一段静かに（絵文字減・断定減・「今日はもう休みましょう」の締め） | 深夜は不安のピーク帯。煽らない配慮が最も試される時間（06 #89とも連動） |
| 送信不能状態の排除 | エラー時も入力内容をローカル保持し、再送導線を出す | 深い相談を書いた直後の消失は致命的な信頼毀損 |

## 5. 演出・感情設計

- **音**: デフォルトはミュート。カードめくり・結果開示の効果音はオプトイン。夜間帯は自動で音無し
- **ハプティクス**: カードをめくる瞬間・重要な結果の開示時のみ軽い振動。多用しない（1セッション3回まで）
- **アニメーション**: 「速い操作は即時、意味のある瞬間だけ遅く」。画面遷移は200ms以内、儀式演出（シャッフル・星盤）だけ3〜5秒かける
- **儀式感**: 深い相談の鑑定前に「ひとつ深呼吸をして、心を静めてください」のプロンプトを挟む（スキップ可）。儀式が結果への構えを作り、体験の密度を上げる
- **色とライティング**: 基調は夜空（深紺・紫・金）。「夜空の下で、いちばん話を聞いてくれる存在」（Spine §9）を視覚言語で体現する。警告・不安を連想させる赤の大面積使用は禁止。時間帯で背景の空が変わる（朝=薄明、夜=星空）

## 6. 課金導線UX

`10_Pricing.md` の価格体系を前提に、見せ方だけを定義する。

| 項目 | 基準 |
|---|---|
| ペイウォールを出すタイミング | **価値実感後のみ**。初回はAha体験後、通常時は鑑定完了直後の「さらに深く視る」文脈（06 #47）。起動直後・鑑定前のペイウォールは禁止 |
| 価格提示の順序 | Premium ¥9,800 → Standard ¥2,980（おすすめバッジ） → Light ¥980 の順（アンカリング、06 #58）。単発チケットは「まず試す」の位置に併記（06 #46） |
| 無料枠の見せ方 | 「今日の無料鑑定 残り1回」の事実表示は可。「急いで！」「今だけ！」等の焦燥煽りテキスト・カウントダウン演出は禁止。残数ゼロ時は「明日また1回回復します」と回復を明示（06 #55） |
| 初回オファー | 初月トライアル価格（06 #45）は初回課金検討画面でのみ提示。リターゲティング的に何度も追いかけない |
| 解約UX | 設定から2タップ以内で解約画面に到達できること。引き留めは「理由1問→理由に応じたオファー1回だけ」（06 #110, #115, #116）。オファー拒否後は即座に解約完了させる。**引き留めすぎないことがブランド保護**であり、円満な解約者は再課金候補（06 #113, #117）である |

## 7. 通知設計

### 通知の哲学

Co-Star が証明したのは「通知文そのものがスクショされSNSの話題になる」こと。ただし同社の不穏で断定的な文体はブランド方針（Spine §9）と両立しない。**本サービスの通知は「詩的で、具体的で、開かなくても少し心が軽くなる一文」**を基準とする。

- 良い例: 「今日のあなたの月は、無理をしない側にいます。」
- 禁止例: 「今日は重大な変化が起きます。確認してください。」（恐怖・煽り・開封強要）

### 運用基準

| 項目 | 基準 |
|---|---|
| 頻度上限 | 1日2通まで（朝の運勢+夜または答え合わせ）。週合計10通以内 |
| 時間帯 | デフォルト朝7:30/夜21:30。開封実績で個別最適化（06 #44）。23時〜7時は原則送らない |
| 許諾タイミング | OS許諾ダイアログは初回起動時に出さない。「明日、今日の占いの答え合わせを届けます」の文脈画面を挟んでから出す（06 #107） |
| プレ許諾画面の役割 | OSダイアログで一度拒否されると回復が難しいため、自前の説明画面でYesが見込める状態を作ってからOSダイアログを表示する |
| オプトアウト | 通知種類別（朝運勢/答え合わせ/イベント）にオン・オフ可能。全体オフでも退会させない |
| 品質基準 | すべての通知文は原則4（煽らない）の文体ガードを通す。汎用文（「お知らせがあります」）は禁止、必ずパーソナライズド |

## 8. 継続導線 — 毎日開きたくなる仕組み（フックモデル）

Nir Eyal のフックモデル（トリガー→行動→可変報酬→投資）で毎日のループを設計する。リテンションループ（Spine §6-4）のUX実装。

| 要素 | 実装 | 対応改善案 |
|---|---|---|
| **外部トリガー** | 朝の運勢通知、Sunday夜の週間占い、答え合わせ通知、ウィジェットの常時視認 | 06 #23, #28, #31, #29 |
| **内部トリガー** | 「不安になったら開く」「迷ったら聞く」の条件反射を、感情語での想起（不安→あの星空の画面）として育てる。CRMコピーも感情トリガー語で統一 | `11_CRM.md` |
| **行動** | 開いてから報酬まで最短2タップ（今日の運勢は起動即表示、カード1枚引きは1タップ） | 06 #32, #40 |
| **可変報酬** | ①毎日変わる運勢と引くまでわからないカード（狩猟型）②「同じ悩みの人」統計・相性結果（部族型）③自分の記録が育つ実感（自己型）。加えて低頻度ランダムで「スペシャルカード」「特別鑑定の当選」を混ぜる — 毎日同じ+時々特別、が飽きを防ぐ | 06 #32, #14, #43 |
| **投資** | 夜のふりかえり記録、感情ログ、鑑定ノートへの保存、鑑定師との関係性（呼び名・文脈記憶）、ストリーク。**使うほど「ここにしかない自分のデータと関係」が増え、次回のトリガー感度が上がる** | 06 #24, #39, #18, #2, #25 |

**煽らない習慣化の境界線**: ストリークは「積み上げの喜び」までで、「切れる恐怖」で駆動しない。お休み券（06 #37）と依存アラート（06 #89）をセットで実装しない限りストリーク機能は出荷しない。

## 9. 紹介導線UX

詳細戦略は `08_Viral_Strategy.md`。ここではUI上の配置基準のみ定める。

- **シェアボタンの位置**: 鑑定結果カードの直下（感情のピーク直後）。設定画面の奥の「友達に教える」は導線として数えない
- **シェアの摩擦**: 結果カード生成→投稿文自動生成→ワンタップでシェアシート、の3ステップ以内（06 #61, #67, #71）。匿名リンク（06 #64）をデフォルトに
- **相性占いへの自然な導線**: 恋愛・人間関係の鑑定結果の末尾に「お相手との相性を視てみますか？」を文脈的に提示（06 #62）。ホームの機能一覧に置くだけでは使われない
- **受け手の体験**: シェアリンクを開いた人は登録なしで即ミニ体験（06 #75）。「友達の結果を見る→自分もやる」までを60秒以内に設計
- **報酬の見せ方**: 招待特典（06 #59, #68）は「あなたに1枚、お友達にも1枚」と互恵で提示。報酬前置きの拡散依頼はブランドを安売りするため、まず体験のシェア、報酬は後から気づく位置に置く

## 10. 退会防止・休眠復帰UX

### 解約フロー（06 #110/#115/#116の画面設計）

1. 解約導線は隠さない（設定→2タップ以内）
2. 理由を1問だけ選択式で聞く
3. 理由別に**1回だけ**オファーを分岐表示: 「高い」→Light/休会提案、「使わない」→休会+毎朝運勢のみ継続、「合わない」→引き留めず感謝+フィードバック依頼
4. 拒否されたら即解約完了。完了画面は「いつでも戻ってこられます。あなたの記録は残しておきます」で締める（データ保持の安心+復帰障壁の除去）

### 休眠復帰

- 復帰メッセージは汎用文禁止。「あなたがいない間の星の動き」（06 #111）等、その人の命式から個別生成した内容のみ送る
- 復帰初回は「おかえり鑑定」（06 #112）。ブランクを責める・数え上げる表現（「30日ぶりですね」）は禁止し、文脈復元（06 #119）で会話を再開する
- 復帰導線のリンク先はホームではなく、1タップで鑑定が始まる画面に直接着地させる

## 11. UX計測 — 判定指標と目標値

各指標は `14_KPI_Dashboard.md` のダッシュボードに常設し、Spine §5のKPIツリーに接続する。

| 指標 | 定義 | 12ヶ月目標 | 上位KPIへの接続 |
|---|---|---|---|
| 初回鑑定到達率 | 初回訪問者のうち鑑定開始まで到達した割合 | 60%以上 | D1継続率45% |
| 初回鑑定完了率 | 鑑定開始者のうち「今日の一歩」まで読了した割合 | 70%以上 | D1継続率45% |
| Time to Aha 達成率 | 初回10分以内にWRS構成行動（保存/シェア/許諾）へ到達した割合 | 50%以上 | NSM（WRS） |
| 登録転換率 | ミニ鑑定開始者のうち登録完了した割合 | 45%以上 | MAU成長 |
| 通知許諾率 | 登録者のうちプッシュ許諾した割合 | 60%以上 | D7継続率25% |
| 通知開封率（朝運勢） | 配信数に対する開封の割合 | 25%以上 | D7/D30継続率 |
| 鑑定保存率 | 鑑定完了セッションのうち保存された割合 | 30%以上 | WRS |
| シェア率 | 鑑定完了セッションのうちシェアされた割合 | 5%以上 | SNS投稿率3%、K-factor 0.25 |
| ペイウォール転換率 | ペイウォール表示者のうち課金開始した割合 | 8%以上 | 無料→有料転換率6% |
| 解約フロー完遂までの摩擦 | 解約希望から完了までの平均タップ数 | 6タップ以下 | NPS 30（ブランド保護） |
| 解約時セーブ率 | 解約フロー進入者のうち休会/ダウングレードを選んだ割合 | 25%以上 | 月次解約率7% |
| 休眠復帰率 | 復帰メッセージ受信30日以内の再訪割合 | 12%以上 | D30継続率15% |
| クラッシュ・エラー遭遇率 | セッションあたりの致命的エラー率 | 0.5%未満 | 全KPIの土台 |

**運用**: 各UX変更は上表のどの指標を動かすかを事前宣言し、`15_Experiment_Backlog.md` の実験として計測する。宣言できない変更は実装しない。

## 12. アクセシビリティと多言語化の考慮

### アクセシビリティ

- テキストコントラスト比 WCAG AA（4.5:1）以上。夜空基調の配色（§5）でも本文の可読性を犠牲にしない
- 全機能をスクリーンリーダーで完遂可能に（カード演出には代替テキスト、結果は読み上げ順序を保証）。音声読み上げ（06 #6）はアクセシビリティ機能を兼ねる
- 演出・アニメーションはOSの「視差効果を減らす」設定に追従し、点滅表現は使わない
- フォントサイズはOS設定に追従（最大200%までレイアウト崩れなし）
- 色だけで意味を伝えない（吉凶をアイコン+テキストで併記）

### 多言語化（36ヶ月目以降の展開に向けた先行設計、Spine §4）

- 文言のハードコード禁止。初期からi18nリソース化し、鑑定プロンプトもロケール別に差し替え可能な構造にする
- 占術の文化差を前提化する: 四柱推命は繁体字圏で主力、西洋占星術は英語圏で主力。占術モジュールは市場別に構成を変えられる設計にする
- 日付・暦（旧暦・節気）・名前の呼び方（敬称文化）をロケール処理に含める
- 文体ガード（06 #88）・危機対応（06 #97）の禁則辞書と相談窓口リストは言語別に整備しなければ展開しない（倫理基準は全市場共通）
