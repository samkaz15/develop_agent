## 1. バイラル総論

### 1.1 基本式と定義

```
K-factor = i × c
  i = 招待数/ユーザー（アクティブユーザー1人が一定期間に発生させる招待・シェア到達数）
  c = 招待CVR（招待・シェアを受け取った人が登録に至る率）
```

- 計測期間は **登録後30日** をユーザー単位の標準ウィンドウとする（全社共通定義。→ §9.1）
- 「招待」には明示的招待（招待リンク送付）と暗黙的招待（シェア画像の閲覧→流入）の両方を含む。ただしダッシュボード上は **明示K / 暗黙K** を分離して計測する（→ §9.2）

### 1.2 Spine目標への分解

Spineの目標は K-factor 12ヶ月0.25 → 36ヶ月0.45（日本一水準0.5+）、紹介経由新規比率 25%→45%。これを i と c に分解する。

| 時点 | K-factor目標 | i（招待数/ユーザー/30日） | c（招待CVR） | 主な達成手段 |
|---|---|---|---|---|
| 6ヶ月 | 0.15 | 0.75 | 20% | 結果カードシェア + 基本招待リンク |
| 12ヶ月 | 0.25 | 1.00 | 25% | 相性占い招待 + 双方向インセンティブ |
| 24ヶ月 | 0.35 | 1.17 | 30% | ペア鑑定・グループ占い・Wrapped初回 |
| 36ヶ月 | 0.45 | 1.29 | 35% | 占いバトン・匿名リンク・Wrapped定着 |

**判定文**: iを1.3以上に無理に伸ばすより、cを35%まで引き上げる方が持続的である。cはランディング体験（招待経由専用LP、相手の名前入り、30秒で初回鑑定）で決まるため、招待流入の初回鑑定到達率80%以上を維持できない施策は差し戻す。

補助分解（12ヶ月時点の月間規模イメージ）:

```
月間アクティブ100万人 × i 1.00 = 招待到達100万件
100万件 × c 25% = 紹介経由新規25万人/月
→ 新規全体が月100万人なら紹介比率25%（Spine整合）
```

### 1.3 原則: バイラルは機能ではなく感情から生まれる

シェアボタンを置いてもシェアは起きない。**「この結果を、あの人に見せたい」という感情が先にあり、機能はそれを摩擦ゼロで通すだけ**である。したがって全バイラル施策は「どの感情を、誰に向けて、どの瞬間に発火させるか」から設計する。

**シェアの動機5類型**（全施策はこのいずれかに必ず紐づける。紐づかない施策は実施しない）:

| # | 動機 | ユーザーの内心 | 対応する本サービスの出力 | 主な発火先 |
|---|---|---|---|---|
| 1 | 自己表現 | 「これは私のことだ。私をわかってほしい」 | 性格・資質の言語化カード、「あなたの本質」系鑑定 | Instagramストーリーズ、X |
| 2 | 共感獲得 | 「わかる〜って言ってほしい」 | あるある系の運勢文、感情を言い当てる一文 | X、TikTokコメント欄 |
| 3 | 有用性提供 | 「あの子に教えてあげたい」 | 相手の星座・悩み向けアドバイス、今日のラッキー情報 | LINE個別トーク |
| 4 | 話題提供 | 「これネタになる」 | 意外性のある診断結果、ランキング、バトン | グループLINE、X |
| 5 | 関係性確認 | 「私たちの関係を確かめたい」 | 相性占い、ペア鑑定、記念日鑑定 | LINE個別トーク、カップル/推し界隈 |

**判定文**: 新規バイラル施策の企画書には「動機類型 / 想定シェア先 / 発火瞬間」の3点を必須記載とし、欠落した企画はレビューで差し戻す（→ [15_Experiment_Backlog.md](./15_Experiment_Backlog.md) のテンプレートに組込済）。

### 1.4 ブランド制約（バイラルの禁じ手）

[09_Brand_Strategy.md](./09_Brand_Strategy.md) のブランド倫理に従い、以下は**K-factorが上がるとしても実施しない**:

- 恐怖・不安を煽るシェア文言（「あなたの運勢がヤバい」等）
- 強制シェア（シェアしないと結果が見られない）— 結果は常にシェアなしで閲覧可能
- 相手の同意なく他人の鑑定結果を公開させる導線
- ネガティブ内容を含むカードの自動シェア（→ §2.2 自動除外ルール）

## 2. シェアしたくなる演出設計

### 2.1 「誰かに見せたい」瞬間の特定

シェア発火の瞬間はプロダクト内で有限個に特定できる。以下を **Share Momentカタログ** として管理し、各瞬間にシェア導線を1つずつ実装・計測する。

| # | 瞬間 | 発火する動機（§1.3） | 導線 | シェア率目標 |
|---|---|---|---|---|
| SM-1 | 鑑定結果で「言い当てられた」と感じた直後 | 自己表現・共感獲得 | 結果カード生成→ストーリーズ共有 | 表示に対し8% |
| SM-2 | 今日の運勢が良かった朝 | 話題提供 | 運勢カードのワンタップ共有 | 3% |
| SM-3 | 相性占いの結果閲覧時 | 関係性確認 | 「相手に送る」ボタン | 25% |
| SM-4 | ストリーク達成（7/30/100日） | 自己表現 | ストリークカード | 10% |
| SM-5 | 月末「今月のあなた」閲覧時 | 自己表現 | まとめカード | 15% |
| SM-6 | Wrapped閲覧時（年1回） | 自己表現・話題提供 | Wrappedカード一式 | 35% |
| SM-7 | 悩み相談で心が軽くなった直後 | 有用性提供 | 「同じ悩みの友達に送る」 | 2%（低くて良い。私的な瞬間のため控えめに） |

**判定文**: SM-7のような私的な瞬間ではシェアを強く促さない。「心が軽くなった直後に営業される」体験はブランドプロミス違反であり、この瞬間の導線は控えめなテキストリンク1つに留める。

### 2.2 結果カードのデザイン要件

シェアされる実体は「画像」である。結果カードは以下を満たすことを実装要件とする。

**フォーマット要件**

| 項目 | 要件 | 判定基準 |
|---|---|---|
| サイズ | 縦9:16（1080×1920）と正方形1:1（1080×1080）の2種を**自動生成** | 生成失敗率0.1%未満、生成時間1.5秒以内 |
| 中心要素 | 鑑定結果から抽出した**名言的な一文**（20〜40字）を最大級のタイポグラフィで | 一文はAIが3候補生成しユーザーが選択可能 |
| ブランド要素 | ロゴ透かし + サービスURL/QRを下部に固定 | 透かし除去オプションは提供しない |
| 日付 | 鑑定日を必ず記載（「2026.07.11の星の言葉」形式） | 日付なしカードは生成しない（後日再シェア時の鮮度混乱を防ぐ） |
| 世界観 | 夜空トーンの背景 + 温かい光のアクセント（[09_Brand_Strategy.md](./09_Brand_Strategy.md) のビジュアル原則準拠） | ブランドカラーパレット外の配色を使用しない |
| 可変性 | 星座・テーマ・季節で背景バリエーションが日替わり（→ §5.1） | 同一ユーザーが7日連続で同一背景にならない |

**内容の安全要件（ネガ内容の自動除外）**

- シェア用カードには、鑑定本文のうち **ポジティブ/中立の文のみ** を転載する。注意・警告・課題に言及する文は自動除外する
- 除外判定は鑑定生成時にAIが文単位でタグ付け（`shareable: true/false`）し、カード生成器は `shareable: true` の文のみ使用する
- 例: 本文「今週は対人運に注意。ただしあなたの聞く力は強い武器になる」→ カードには「あなたの聞く力は、強い武器になる」のみ
- **判定文**: 誤ってネガ文がカードに載った事例は Sev2 バグとして扱い、48時間以内に修正する（本人の自己開示範囲を超えてネガ情報が拡散するのはブランド毀損リスク。→ 09 §リスク対応）

### 2.3 スクショ検知→シェア画像提案

- アプリがスクリーンショット撮影を検知したら、トースト表示: 「きれいなシェア用カードもあるよ 🌙」→ タップで2フォーマットのカードを即時提示
- 目的: (a) 見切れた汚いスクショではなくブランド透かし入りカードを流通させる、(b) スクショ＝シェア意図のシグナルとして計測する
- 計測: スクショ検知数 / 提案表示→カード保存率（目標40%以上）/ そのままシェア率
- **やらないこと**: スクショの禁止・警告表示。撮影は自由であり、提案はあくまで「もっといいものがある」の提示

### 2.4 シェア導線のUX基準

| 基準 | 数値 |
|---|---|
| 結果表示→シェア完了までのタップ数 | 3タップ以内 |
| シェア先の並び順 | LINE → Instagramストーリーズ → X → 画像保存（日本のコア層の利用実態順。ABテストで最適化） |
| カードプレビュー表示までの待ち時間 | 1.5秒以内（超える場合はスケルトン表示） |
| シェア後の体験 | 「シェアありがとう」+ 招待リンクの自動添付案内（押しつけない1画面のみ） |

## 3. 構造的バイラル機能

シェア（暗黙的バイラル）に加え、**機能の構造自体が他者を必要とする** 明示的バイラルを設計する。優先度順に記載する。

### 3.1 相性占い（最優先・リファラルループの心臓）

**設計**: 2段階開示モデル

1. **簡易版**: 相手の生年月日（+任意で名前）だけで即座に相性を占える。登録不要の相手情報でOK
2. **完全版**: 「相手を招待して、相手も鑑定に答えると **ふたりの相性の全章が解放**」— 相手側の性質・すれ違いポイント・関係を深める行動提案が追加される

| 項目 | 内容 |
|---|---|
| 動機類型 | 関係性確認（+有用性提供） |
| 招待の必然性 | 完全版は構造的に相手の参加が必要（人工的なゲートではなく、情報として本当に相手の回答が要る建て付けにする） |
| 招待文面 | 「〇〇との相性、途中まで占ったよ。あなたが答えると全部見られるみたい」— 自動生成しつつ編集可能 |
| 対象関係 | 恋愛 / 片思い / 友達 / 仕事仲間 / 家族（恋愛限定にしない。シェア母数が5倍になる） |
| KPI | 簡易版実施者の招待送付率 30% / 被招待者の登録率 40%（関係性招待は一般招待よりcが高い） |
| 課金接続 | 相性の深掘りは Standard（¥2,980）の「相性無制限」に接続（→ [10_Pricing.md](./10_Pricing.md)） |

### 3.2 ペア鑑定（恋人 / 友達 / 推し）

- ふたりで受ける前提の鑑定パッケージ。「今月のふたりの過ごし方」「ふたりの記念日運勢」
- **推しペア鑑定**: 相手が実在の知人でなくても成立する変種（推しの公開プロフィール生年月日を使用）。Z世代・推し活文脈のUGC発生源として設計（動機類型: 自己表現+話題提供）。結果カードはファンアート的に装飾可能
- KPI: ペア鑑定実施ペアの両者D7継続率 45%以上（単独ユーザー比+20pt を仮説として検証）

### 3.3 グループ占い（今日のチーム運勢）

- 3〜8人のグループ（職場チーム・友達グループ・部活）でリンクを共有すると、毎朝「今日のチーム運勢 + 今日のキーパーソン」が届く
- 配信先はグループLINEを想定（LINEミニアプリ連携。→ [11_CRM.md](./11_CRM.md)）
- バイラル構造: グループ作成者1人が2〜7人を自動的に巻き込む。i への寄与が最も大きい機能候補
- KPI: グループ作成→2人以上参加率 70% / グループ参加者の単独利用転換率 25%

### 3.4 占いバトン

- 「この鑑定、次はあなたの番」— 鑑定テーマ（例: 「あなたの2026下半期の隠れ資質」）を指名制で回すSNSネイティブ機能
- 指名は3人まで。バトンカードに指名相手の名前（またはSNSハンドル）が入る
- 季節イベント（→ §7）と連動して四半期に1回、テーマを新調して投下する
- 動機類型: 話題提供+関係性確認。X・Instagramストーリーズの「バトン文化」に接続

### 3.5 匿名リンク「あなたを占ってもらいたい人がいます」

- ユーザーが匿名のまま、特定の相手に鑑定を贈れるリンク。受け手には「あなたのことを想っている誰かから」とだけ表示
- 用途: 直接は言えない相手への気遣い（落ち込んでいる友達、頑張っている同僚）
- **安全設計（必須）**: (a) ネガティブな鑑定テーマは匿名贈付不可、(b) 受け手は差出人推測ゲームを強要されない、(c) 嫌がらせ利用の通報導線と同一受け手への送付回数制限（同一受け手に月3通まで）、(d) 受け手がオプトアウトすると以後受信しない
- 動機類型: 有用性提供。開封率目標 60%（「誰かが自分を想っている」は最強の開封動機）

### 3.6 機能優先度まとめ

| 機能 | 実装優先度 | 期待K寄与 | 実装コスト | 投入時期目安 |
|---|---|---|---|---|
| 相性占い2段階開示 | P0 | 高 | 中 | 〜3ヶ月目 |
| 結果カード自動生成（§2） | P0 | 高 | 中 | 〜3ヶ月目 |
| 招待プログラム（§4） | P0 | 中 | 低 | 〜3ヶ月目 |
| グループ占い | P1 | 高 | 高 | 〜9ヶ月目 |
| ペア鑑定 | P1 | 中 | 中 | 〜9ヶ月目 |
| Wrapped（§5.4） | P1 | 高（季節性） | 高 | 初年度12月に間に合わせる |
| 占いバトン | P2 | 中 | 低 | 〜12ヶ月目 |
| 匿名リンク | P2 | 中 | 中 | 安全設計確定後 |

## 4. 招待プログラム

### 4.1 双方向インセンティブ

| 対象 | 報酬 | 付与条件 |
|---|---|---|
| 招待側 | 深掘り鑑定チケット1枚（¥300相当。→ [10_Pricing.md](./10_Pricing.md)） | 被招待者が登録し**初回鑑定を完了**した時点 |
| 被招待側 | 深掘り鑑定チケット1枚 + 初回体験の拡張（通常より1テーマ多く無料鑑定） | 登録完了時点 |

設計原則:
- 報酬は**現金・割引ではなくプロダクト内価値（チケット）**で払う。理由: (a) 限界コストが低い、(b) 報酬消化がそのまま価値体験になり継続に効く、(c) 換金性がなく不正の旨味が薄い
- 付与条件を「登録」ではなく「初回鑑定完了」に置く。理由: 空アカウント量産の抑止と、cの質（アクティベーション済み新規）の担保

### 4.2 マイルストーン報酬

| 累計招待成功数 | 報酬 | 意図 |
|---|---|---|
| 1人 | チケット1枚（基本報酬） | 最初の1人の心理障壁を下げる |
| 3人 | 限定鑑定「あなたと友情の星回り」解放 + チケット2枚 | 3人=グループLINE1つ分。ここを第一目標に置く |
| 10人 | Standardプラン1ヶ月無料 + 限定プロフィール称号「星をつなぐ人」 | ヘビー招待者の顕彰。金銭報酬より称号が効く層 |

- 上限: 報酬付与対象は月10人まで（それ以上の招待は歓迎するが報酬は付かない。業者的招待の抑止）

### 4.3 招待文面のパーソナライズ

招待の c は文面で大きく変わる。デフォルト文面を状況別にAI生成し、ユーザーが編集して送る。

| 文脈 | 生成される文面例 |
|---|---|
| 相性占いから | 「〇〇と私の相性占ってみたら途中までしか見れなくて。あなたが生年月日入れると続き見られるらしい」 |
| 結果カードから | 「これ私の今日の占いなんだけど、当たりすぎて笑った。あなたのも見てみて」 |
| 汎用招待 | 「最近使ってる占い、脅してこないし普通に話聞いてくれて良い。このリンクからだと1回多く占える」 |

- ルール: 文面に**誇張・恐怖訴求を含めない**（「絶対当たる」「やらないと損」は禁句。→ 09 トーン&ボイス）
- 招待リンクは個人固有 + 文脈パラメータ付き（どの導線からの招待かをc計測に使う）

### 4.4 不正対策

| リスク | 対策 | 判定基準 |
|---|---|---|
| 自己招待（複数アカウント） | デバイスフィンガープリント + 電話番号/LINE ID重複検知。同一デバイスからの被招待登録は報酬対象外 | 検知時は報酬没収、通知は「条件を満たしませんでした」に留める |
| 招待リンクのばら撒き（掲示板・リワードサイト） | 同一リンクからの登録が24時間で20件を超えたら自動レビュー、報酬保留 | レビューで承認/没収を48時間以内に判定 |
| 空アカウント量産 | 報酬条件を「初回鑑定完了」に設定済（§4.1）+ 完了までの行動パターン異常検知 | 完了まで30秒未満のセッションは報酬対象外 |
| 報酬目当ての解約再登録 | 被招待報酬は電話番号/LINE IDごとに生涯1回 | — |

- 不正判定の運用: 誤検知時の問い合わせにはCSが24時間以内に回答。**ブランド上、疑わしきはユーザー有利に倒す**（グレー判定は付与）

## 5. 毎日投稿したくなる仕組み（習慣型バイラル）

リテンションループ（Spine §6-4）とUGCループ（§6-2）を接続する。「毎日使うもの」が「ときどきシェアされるもの」になる設計。

### 5.1 今日の運勢カードの日替わりデザイン

- 背景アートを**365日日替わり**にする（月相・季節・二十四節気を反映した夜空アート）。「今日のカードは今日しか手に入らない」収集性がシェアと再訪の両方を駆動する
- 週1回「特別背景の日」（新月・満月・節気）を設定し、CRM通知と連動（→ [11_CRM.md](./11_CRM.md)）
- 計測: 特別背景日のシェア率が通常日比 +50% を下回ったらデザイン敗北と判定し翌月改訂

### 5.2 ストリーク自慢

- 連続利用日数を美しいカード化（7日=三日月、30日=半月、100日=満月、365日=一年の星図）
- ストリーク断絶の扱い: **「途切れた」を責めない**。「おかえり」設計とし、月1回の「星の休息日」（1日欠けてもストリーク維持）を付与。恐怖・焦燥でストリークを維持させない（依存誘発の禁止。→ 09）
- シェア導線は達成日にのみ表示（毎日は出さない）

### 5.3 月末「今月のあなた」まとめカード

- 毎月末日〜翌月3日に生成: 今月の相談テーマ分布 / 心が動いた鑑定の一文 / 来月のテーマ
- 動機類型: 自己表現。「月報を自分のストーリーズに上げる」文化を作る
- シェア率目標 15%（SM-5）。Wrapped（§5.4）の月次縮小版として同一データ基盤で実装し、Wrappedの品質を年12回リハーサルする位置づけ

### 5.4 年次Wrapped型イベント（12月の最重要バイラルイベント）

**位置づけ**: 年間で最大のバイラル瞬間。Spotify Wrappedが証明した「1年の自己表現需要」を占い文脈で取りにいく。**12月第1週公開**（Spotify Wrappedと同時期の話題性に相乗りしつつ、年末の内省需要のピーク前に置く）。

**コンテンツ構成（スライド式・全8枚）**

| # | スライド | 内容 | シェア適性 |
|---|---|---|---|
| 1 | 表紙 | 「あなたの2026年の星の記録」+ 利用日数 | 中 |
| 2 | **今年の感情曲線** | 12ヶ月の相談トーンから推定した感情の波形グラフ。「4月、あなたはいちばん揺れていた」 | 高（最強の自己表現素材） |
| 3 | **相談テーマTop3** | 恋愛42% / 仕事31% / 自分のこと27% のような構成比 | 高 |
| 4 | いちばん心が動いた言葉 | 保存・再読された鑑定文からAIが選ぶ「今年のあなたへの一文」 | 高 |
| 5 | あなたの成長 | 年初と年末の相談の変化（「聞き方が変わった」を言語化） | 中 |
| 6 | 星の統計 | 鑑定回数・最長ストリーク・いちばん多かった時間帯 | 中 |
| 7 | **来年のテーマ** | 2027年のあなたのキーワード1語 + 短い展望 | 高（「来年の私」は年始の話題素材） |
| 8 | 締め | 「来年も、夜空の下で待ってるね」+ 招待リンク | — |

**実装・運用要件**

- 全スライドが9:16カードとして個別保存・シェア可能。ネガ内容の自動除外ルール（§2.2）適用。感情曲線の「揺れ」表現は否定的でなく「それだけ真剣だった」の文脈で書く
- 対象: 年間10セッション以上のユーザー（それ未満には簡易版「あなたの星の芽」を出し、疎外感を作らない）
- 非ユーザー向け: Wrapped閲覧ページから「あなたの2027年テーマだけ先に占う」導線（被招待CVR装置）
- 負荷設計: 事前バッチ生成 + 公開日のスパイク（通常比10倍想定）に耐えるキャッシュ設計
- KPI: 対象者の閲覧率 70% / 閲覧者のシェア率 35%（SM-6）/ Wrapped経由新規が12月新規の20%
- 事後検証: 1月第2週にレビュー会を実施し、翌年版のバックログを起票（→ [16_Growth_Operating_System.md](./16_Growth_Operating_System.md)）

## 6. 口コミ設計

バイラルは画像だけでなく「会話」でも起きる。口コミは自然発生を待たず、**発生確率を設計**する。

### 6.1 NPS計測→推奨者への口コミ依頼フロー

```
NPS調査（四半期 + 課金後30日時点）
  ├── 推奨者（9-10）→ 48時間以内に「口コミ依頼」フロー
  │     ① 感謝メッセージ
  │     ② 「よかったら、誰かに教えてあげてください」+ パーソナライズ招待文面（§4.3）
  │     ③ App Storeレビュー依頼（未依頼者のみ）
  ├── 中立者（7-8）→ 依頼せず。改善要望の自由記述のみ収集
  └── 批判者（0-6）→ 依頼せず。CSフォローアップ（→ 11_CRM）
```

- **判定文**: 中立者・批判者にレビュー依頼を出す実装は禁止（低評価レビューの自己生産になる）
- 目標: 推奨者の口コミ依頼→招待送付率 20%

### 6.2 App Storeレビュー依頼のタイミング

原則: **感情のピーク直後、かつ邪魔にならない瞬間**にのみ出す。

| 出す瞬間（いずれか初回） | 出さない瞬間 |
|---|---|
| 鑑定結果を保存した直後 | 鑑定の途中・チャットの最中 |
| ストリーク7日達成の翌回起動時 | 課金直後（「金を払わされた直後」は感情が割れる） |
| 相性占い完全版を見終わった直後 | 解約・エラー・ネガ寄り鑑定の直後 |
| NPS 9-10 回答直後 | 初回セッション（早すぎる） |

- 頻度: OS標準のレビューダイアログ制限に従い、独自プロンプトは90日に1回まで
- 目標: ストア評価 4.5以上を常時維持。4.3を下回ったら依頼ロジックとプロダクト品質の緊急レビューを起動

### 6.3 「言いやすいひとこと」の提供（口コミの台詞をプロダクトが用意する）

口コミが起きない最大の理由は「なんて説明すればいいかわからない」。**プロダクトが台詞を配る**。

| シーン | 提供する台詞例 |
|---|---|
| 友達に聞かれた時 | 「占いっていうか、夜に話聞いてくれるAI。全然脅してこないのが良い」 |
| SNSで紹介する時 | 「不安を煽ってこない占い、初めてかもしれない」 |
| 悩んでる友達に | 「これ、答えくれるっていうより整理してくれる感じ。無料でできるよ」 |

- 実装: 招待画面・シェア画面に「こんなふうに紹介してね」として2〜3案を常時表示。コピーは四半期ごとにABテストで改訂
- この台詞群はブランドの一行紹介（→ 09 ポジショニング）と同一の骨格を持たせ、口コミがそのままブランドメッセージになる状態を作る

## 8. バイラルとファネルの接続

バイラルは新規獲得で終わらない。被招待者の体験を分解して管理する。

```
招待到達 → LP閲覧 → 登録 → 初回鑑定完了 → D7残存 → （自らが招待者になる）
```

| ステップ | 目標転換率 | 責任ドキュメント |
|---|---|---|
| 招待到達→LP閲覧 | 60% | 本書（文面・§4.3） |
| LP閲覧→登録 | 45%（招待専用LP: 招待者名入り・30秒で体験開始） | 本書 + [07_UX_Improvement.md](./07_UX_Improvement.md) |
| 登録→初回鑑定完了 | 80% | [07_UX_Improvement.md](./07_UX_Improvement.md) |
| 初回完了→D7残存 | 30%（一般新規比+5pt を要求。関係性文脈で入った新規は残りやすい前提を検証） | [11_CRM.md](./11_CRM.md) |
| D7残存→30日以内に自ら招待 | 20% | 本書（バイラルの複利条件） |

**判定文**: 「D7残存→自ら招待」が15%を下回る場合、バイラルは複利にならず一過性キャンペーンに退化していると判定し、報酬設計（§4）より先にShare Moment（§2.1）の体験品質を疑う。

## 9. 計測とダッシュボード

### 9.1 指標定義（正本）

| 指標 | 定義 | 計測粒度 |
|---|---|---|
| シェア率 | シェア導線表示UUに対する、シェア完了（外部アプリ起動 or 画像保存）UUの比率 | Share Moment別 × 日次 |
| 招待率 | 週次アクティブUUに対する、招待リンク送付UUの比率 | 週次 |
| i（招待数/ユーザー） | 登録後30日間の、明示招待送付数 + 暗黙シェア到達推定数 / 該当コホートUU | 月次コホート |
| c（被招待CVR） | 招待リンク/シェア流入UUに対する登録完了率 | 導線別 × 週次 |
| K-factor | i × c（明示K / 暗黙K / 合算Kの3系統） | 月次コホート |
| 紹介経由新規比率 | 新規登録のうち、招待リンク・シェア流入・紹介コード経由の比率 | 月次 |
| SNS投稿率 | 月間の投稿UU（トラッキング可能なシェア完了UU）/ MAU（Spine §5: 12ヶ月3%→36ヶ月6%） | 月次 |

計測上の注意:
- 暗黙シェア（画像保存→手動投稿）はリンクを踏まない流入を生むため、**カード内QR/URL経由流入 + 流入時アンケート「どこで知った?」** で補正係数を四半期ごとに更新する
- 招待CVRの分母は「リンククリック」ではなく「リンク到達（プレビュー表示含む）」を採用できる範囲で広く取り、導線間で定義を統一する

### 9.2 ダッシュボード構成（→ [14_KPI_Dashboard.md](./14_KPI_Dashboard.md) に実装）

| ビュー | 内容 | 更新 | 閲覧者 |
|---|---|---|---|
| Viral Overview | K-factor（明示/暗黙/合算）、紹介経由比率、対Spine目標の進捗 | 日次 | 全社 |
| Share Moments | SM-1〜7別のシェア率・トレンド・ABテスト状況 | 日次 | Growth |
| Invite Funnel | §8のファネル5段の転換率、導線別c | 日次 | Growth |
| Fraud Monitor | 不正検知件数・保留報酬・誤検知率 | 日次 | Growth + CS |
| Event Tracker | イベント期間中の専用ビュー（目標対比リアルタイム） | 時間次 | 全社（イベント期間のみ） |

### 9.3 改善サイクル

- **週次**: Share Moment別シェア率と導線別cをレビュー。最も目標乖離が大きい1点に翌週の実験を集中（→ 16 運用OS）
- **月次**: コホートK-factorを確定し、Spine目標ラインとの差分を i / c に帰着させて要因分解。実験バックログ（→ 15）の優先度を更新
- **四半期**: 暗黙シェア補正係数の更新、招待報酬の経済性レビュー（報酬コスト/紹介経由新規1人 が CAC目標 ¥1,500（12ヶ月時点）を上回っていないか検証）
- **実験の最低品質基準**: 全バイラル実験は (a) 動機類型の明記、(b) 主要指標と最小検出効果、(c) ブランド制約チェック（§1.4）通過、の3点を満たすこと

### 9.4 撤退基準

| 施策 | 撤退/改訂の判定条件 |
|---|---|
| 招待報酬 | 報酬コスト/紹介新規1人 > ¥1,500 が2ヶ月連続 → 報酬額または条件を改訂 |
| 各Share Moment | シェア率が目標の1/3未満が4週継続 → 導線を撤去しUX負債化を防ぐ |
| バイラルイベント | 新規増分がイベントコスト（開発+運用）換算CAC ¥3,000 超 → 翌年は縮小または廃止 |
| 匿名リンク | 通報率が送付数の0.5%超 → 即時停止し安全設計を再審査 |

外部数値は推定であり断定しない（Spine §11）。設計判断の参照点として記録する。
