## 1. ファイル出力規則

### 1.1 出力先パス

- ユーザー指定がない場合の出力先: `docs/Reports/YYYY-MM-DD_{slug}.md`（SKILL.md Step 10 と同一）
- `YYYY-MM-DD` は**作成日**（メタ情報表の作成日と一致させる）
- `docs/Reports/` が存在しない場合は作成する。ユーザーが別パスを指定した場合はそちらを優先し、ファイル名規則（1.2）だけは維持する

### 1.2 ファイル名slug規則

- slugは**英語のケバブケース**: 小文字英数字とハイフンのみ（正規表現: `^[a-z0-9]+(-[a-z0-9]+)*$`）
- 長さ: 3〜6語、50文字以内
- 内容がわかる名詞句にする（レポート種別 + 対象）。例:
  - ✅ `edtech-market-entry-analysis`
  - ✅ `competitor-analysis-saas-crm`
  - ❌ `report1`（内容不明）
  - ❌ `市場調査`（日本語禁止）
  - ❌ `EdTech_Market_Report`（大文字・アンダースコア禁止）
- 同日同slugの既存ファイルがある場合は `-v2`, `-v3` を末尾に付ける（上書き禁止）

### 1.3 1レポート1ファイル

- 1つのレポートは**1つの.mdファイルで完結**させる。分冊・別添ファイルへの分割は禁止
- 長大な補足データは同一ファイル内の `<details>`（§2.4）に格納する
- レポートから他ファイルへの相対リンクは、リポジトリ内に実在するファイルのみ許可（§10 リンク切れ禁止）

## 2. Markdown/GitHub最適化

### 2.1 GFM準拠

- 方言はGFM（GitHub Flavored Markdown）に限定する。GFMでレンダリングされない記法（Setext見出し、定義リスト、LaTeX数式ブロック `$$`）は使用禁止
- 改行は「空行で段落分け」を基本とする。行末2スペースによる強制改行は禁止（不可視で保守不能）
- インデントによるコードブロック化（4スペース）は禁止。コードブロックは必ずフェンス（```）で作る

### 2.2 表のパイプ整形

- 表の各行は先頭と末尾にパイプを置く: `| セル | セル |`
- ヘッダ区切り行は最低3ハイフン: `|---|` 以上。数値列は `|---:|`（右揃え）
- ソース上の列幅揃え（スペースパディング）は**行わない**（日本語の全角文字幅で揃わず、差分も汚れる。レンダリング結果はパディングと無関係）
- セル内でパイプ文字が必要な場合は `\|` とエスケープする
- セル内改行が必要な場合のみ `<br>` を許可（1セル1回まで。2回以上必要なら表設計を見直す）

### 2.3 アンカーリンクと目次生成

GitHubの見出しアンカーは次の規則で生成される。目次リンクはこの規則で書く。

1. 見出しテキストを小文字化（英字のみ影響）
2. 半角スペースをハイフンに置換
3. ピリオド・カッコ等の記号を削除（日本語・英数字・ハイフンは残る）

| 見出し | アンカー |
|---|---|
| `## 1. Executive Summary` | `#1-executive-summary` |
| `## 8. Strategic Recommendations` | `#8-strategic-recommendations` |
| `### 8.1 戦略オプション比較` | `#81-戦略オプション比較` |

- 目次生成規則: H2が4章以上なら目次必須（`STYLE_GUIDE.md` §10.3）。目次は番号付きリストで、H2のみ掲載
- 出力前検証: 目次の全リンクについて、リンク先アンカーが本文見出しから上記規則で導出した文字列と一致することを確認する（不一致1件でもStep 11不合格）

### 2.4 折りたたみ `<details>` の使用場面

`<details>` を使ってよいのは以下の2場面**のみ**:

1. **補足データ**: 本文の結論に直接使わないが検証可能性のために残す生データ・計算過程
2. **長い表**: 10行超の表の全量版（本文には「上位N + その他」の要約表を置いたうえで格納する。`STYLE_GUIDE.md` §4.1）

書式（この形式に固定する）:

```markdown
<details>
<summary>補足: {{内容がわかるタイトル}}（クリックで展開）</summary>

{{内容。前後に空行を必ず入れる — 空行がないとMarkdownがレンダリングされない}}

</details>
```

- 禁止: 本文の結論・推奨・リスクを `<details>` に入れること（畳まれて読まれない）。1レポートに `<details>` は5個まで

## 3. Mermaid規定

### 3.1 使用可能な図種

以下の5種**のみ**使用可（`STYLE_GUIDE.md` §5.1 と整合）。

| 図種 | 用途 | 主な使用章 |
|---|---|---|
| `flowchart` | フロー・因果・KPIツリー | 9. KPI |
| `gantt` | ロードマップ | 10. Roadmap |
| `quadrantChart` | 2軸ポジショニング | 5. Competitor Analysis |
| `pie` | 構成比（5項目以内） | 4. Market Research |
| `timeline` | 時系列の経緯 | 3. Background |

`mindmap`, `erDiagram`, `sequenceDiagram`, `classDiagram` 等の上記以外の図種は禁止（GitHubのMermaidバージョンによりレンダリングが不安定なもの・レポート用途に不要なものを除外）。

### 3.2 レンダリング失敗を防ぐ規則

1. **日本語ラベルは必ずダブルクォートで囲む**: `A["市場規模 3,200億円"]`（クォートなしの日本語・記号はパースエラーの主因）
2. ラベル内で使用禁止の文字: `"`（ラベル内では `#quot;` を使う）、`(` `)` `[` `]` `{` `}`（クォート内でも避け、全角括弧 `（）` に置換する）、`;`、改行
3. ノードIDは半角英数字のみ（`A`, `B1`, `kpi2`）。日本語IDは禁止（ラベルに日本語を書く）
4. `%%` コメント行はブロック先頭のみ許可（行末コメントはパースエラーの原因）
5. `gantt` の日付は `dateFormat YYYY-MM-DD` を宣言し、タスク行の日付もこれに従う
6. `quadrantChart` の軸ラベル・タイトルも日本語はクォートで囲む
7. 1つのMermaidブロック内で図種の混在は不可（1ブロック1図）
8. 複雑度上限（ノード12個・3階層等）は `STYLE_GUIDE.md` §5.2 に従う

### 3.3 フォールバック（表に落とす）

以下のいずれかに該当したら、Mermaidをやめて表で表現する:

- 複雑度上限（`STYLE_GUIDE.md` §5.2）のいずれかを超える
- §3.2 の規則で表現しきれないラベル・構造がある
- 図にしても表以上の情報（構造・流れ・位置関係）が加わらない

フォールバック時の対応表:

| 元の図 | 代替の表 |
|---|---|
| flowchart（KPIツリー） | 「階層列付きKPI表」: 階層（L1/L2/L3）列 + 親指標列を持つ表 |
| gantt | フェーズ表（`TEMPLATE.md` §10.1） |
| quadrantChart | 2軸評価表（X軸評価・Y軸評価の2列 + 象限名列） |
| pie | 構成比表（%列 + 合計100%行） |
| timeline | 年表（年月列 + 出来事列） |

## 4. 章番号・見出し規則

`TEMPLATE.md` および `STYLE_GUIDE.md` §1 と完全に同一の規則を適用する（本節は出力検証用の再掲）。

- H1は文書タイトル1つのみ。H2 = 章、H3 = 節。H4は施策見出し（`#### 施策N: 名称`）のみ許可、H5以下禁止
- H2書式: `## {番号}. {英語章名}`（例: `## 7. Key Findings`）。番号 + 半角ピリオド + 半角スペース
- H3書式: `### {章番号}.{節番号} {節名}`（例: `### 7.1 発見の一覧`）。節番号後ろにピリオドなし
- 章を省略した場合は番号を1から詰めて振り直す（欠番禁止）。振り直しは「前提条件」に明記
- 出力前検証（機械的判定）:
  - `#` 開始行が1行のみ
  - `##` 開始行がすべて `## \d+\. ` にマッチする
  - `###` 開始行がすべて `### \d+\.\d+ ` にマッチする（Key Findings の `### F1.` 形式のみ例外）
  - `####` 開始行がすべて `#### 施策\d+: ` にマッチする

## 5. 脚注

### 5.1 記法

- GFM脚注記法を使う: 本文 `[^1]`、定義 `[^1]: 内容`
- 番号は本文の出現順に1から連番（欠番・重複禁止）
- 定義はすべて文書末尾の「参考文献・出典」セクション（§6）にまとめて置く（章の途中に定義を書かない）
- 脚注マーカーは数値・固有の事実の**直後**に付ける: `市場規模は3,200億円[^1]である。`（文末ではなく対象の直後）

### 5.2 脚注に入れるもの / 入れないもの

| 入れるもの ✅ | 入れないもの ❌ |
|---|---|
| 出典（媒体名・タイトル・URL・参照日） | 結論・推奨・リスク（本文に書く） |
| 推定の計算根拠・算出式 | 意思決定に必要な条件・制約（本文に書く） |
| 用語の学術的補足（本文の流れを止めるもの） | 本文の言い換え・繰り返し |
| データの制約（サンプル数・調査時期の注意） | 1文で本文に入る短い補足 |

判定文: **脚注を全部削除しても、レポートの結論・推奨・リスクの理解が変わらないこと**。変わるなら、その脚注の内容は本文に昇格させる。

### 5.3 出典タグとの関係

Step 2（調査）で全事実に付けた出典タグは、Step 10で本節の脚注形式に変換する。外部データで脚注のないものが1件でもあれば品質基準8違反として Step 11 で不合格とする。

## 6. 参考文献・出典

### 6.1 セクションの位置と書式

- 文書の最終セクションとして `## 参考文献・出典` を置く（章番号は付けない。12章の後、`---` 区切りの下）
- 各エントリは脚注定義形式で、以下の5要素をこの順で書く:

```markdown
[^1]: {{媒体名}}「{{タイトル}}」{{発行年}}。{{URL}}（参照: {{YYYY-MM-DD}}）
```

| 要素 | 規則 |
|---|---|
| 媒体名 | 発行組織・メディア名。正式名称 |
| タイトル | 鉤括弧「」で囲む |
| 発行年 | 西暦4桁。月まで判明していれば `2026年3月` |
| URL | 実在確認済みのもののみ。URLがない資料（書籍・社内資料）は「URLなし・{{入手方法}}」 |
| 参照日 | 実際にアクセスした日付（YYYY-MM-DD） |

### 6.2 出典が曖昧な場合の表記

- 出典が特定できない・数値が推定の場合は、脚注に `推定:` プレフィックスで根拠を書く:

```markdown
[^3]: 推定: {{A社公開の会員数N万人}}[^1] に {{業界平均課金率N%}}[^2] を乗じて算出。
```

- **偽の出典URL・実在しない資料名の生成は絶対禁止**（SKILL.md Step 2）。確認できないものは「推定」と書くのが唯一の正しい対応
- 「◯◯と言われている」「一般に」を出典代わりに使うのは禁止。出典が出せないなら推定表記にする

### 6.3 一次情報の記載

- 自社データ・独自調査は媒体名の代わりにデータソースを書く: `[^4]: 自社GA4データ（集計期間: 2026-01-01〜2026-06-30、抽出日: 2026-07-10）`
- インタビュー・ヒアリングは対象と実施日を書く（個人名は書かない）: `[^5]: ユーザーインタビュー（対象: 既存顧客5社、実施: 2026-06）`

## 7. メタ情報

### 7.1 記載位置と書式

| メタ情報 | 位置 | 書式 |
|---|---|---|
| 作成日・調査期間・作成者・対象読者・機密区分・前提条件 | 文書冒頭のメタ情報表 | `TEMPLATE.md` §レポートメタ情報ブロックの8項目表 |
| データ基準日 | 該当する表・数値の直後 | `（2026-06時点）` を表キャプションまたは列ヘッダに付記 |
| 推定・前提の注意 | 該当章の末尾 | `> **注:**` 型（`STYLE_GUIDE.md` §9.1） |
| 免責 | 11. Risks の末尾 | `> **免責:**` 型 |

### 7.2 データ基準日の規則

- 時点によって変わる数値（市場規模・シェア・価格・ユーザー数）には、**表単位で**基準日を明記する（全数値への個別付記は不要、表に1つでよい）
- レポート内で最も古いデータ基準日が作成日から12ヶ月超の場合、鮮度リスクを 11. Risks に必ず1行入れる（SKILL.md §3 エラーハンドリング準拠）

### 7.3 免責の標準文型

```markdown
> **免責:** 本レポートの推定値は {{基準日}} 時点の公開情報に基づく。{{前提}} が変化した場合、
> 結論の見直しが必要である。本レポートは {{対象読者}} の意思決定支援を目的とし、成果を保証するものではない。
```

## 8. 推奨事項の書式

`8. Strategic Recommendations` の各推奨は、以下の**5点セット**を必ず揃える（1つでも欠けたらStep 11不合格）。書式は `TEMPLATE.md` §8.2 の施策表に固定する。

| 項目 | 書き方の規定 |
|---|---|
| 推奨内容 | 動詞で終わる実行文。「◯◯を検討する」は禁止（検討は推奨ではない）。「◯◯を実施する/開始する/停止する」 |
| 根拠 | Key Findings の番号（F1等）への参照を必ず含む。Findingにない根拠を書く場合はまず7章に追加する |
| 期待効果 | 数値 + 期限（例: 「12ヶ月目までにMRR +800万円」）。数値化不能なら代理指標を置き `（推定）` を付ける |
| リスク | 主要リスク1つ以上 + 軽減策。11. Risks の該当番号（R1等）と相互参照する |
| 優先度 | ICE/RICEスコア（Step 7）と順位。スコアなしの「高/中/低」だけは不可 |

- 推奨は優先度順に並べる（スコア降順）
- 「推奨しない案」も8.1の比較表に残す（SKILL.md Step 6: 経営層は選択肢の比較を求める）

## 9. 数値表記統一

レポート内のすべての数値は以下に統一する。1件でも揺れがあれば品質基準9（表記ゆれゼロ）違反。

### 9.1 数値・単位

| 項目 | 規則 | 例 ✅ | 例 ❌ |
|---|---|---|---|
| 桁区切り | 4桁以上の整数はカンマ | 12,500 | 12500 / 1万2500 |
| 金額 | 億円/万円。兆超は兆円。1つの表内で単位統一 | 3,200億円 / 0.5億円 | 32,000千万円 / ¥320B |
| 人数 | 万人/人。1つの表内で統一 | 45万人 | 450,000人と45万人の混在 |
| 割合 | 半角% 、小数第1位まで | 12.3% / 45% | 12.34% / 12.3％（全角） |
| 増減 | 符号を明示。減少は ▲ または - のどちらかに文書内統一 | +12% / ▲3% | 12%増 / マイナス3% |
| 期間 | 「Nヶ月目」（起点はロードマップ開始月）。範囲は「N〜Mヶ月目」 | 3ヶ月目 / 4〜6ヶ月目 | 3か月目 / 3ヵ月後 |
| 日付 | YYYY-MM-DD。月までなら YYYY-MM | 2026-07-11 | 2026/7/11 / 7月11日 |
| 年度 | 「YYYY年」（暦年）/「FYYYYY」（会計年度）を混在させない | 2026年 | 26年 / R8年 |
| 通貨換算 | 外貨は原値 + 換算値 + レート脚注 | 10億ドル（約1,450億円[^n]） | 10億ドル（約1,450億円）のみ |
| 数値レンジ | 全角チルダではなく「〜」（波ダッシュはU+301Cに統一） | 3〜5社 | 3~5社 / 3-5社 |

### 9.2 概数と推定

- 概数は「約」を付ける: `約3,200億円`（元データが概数の場合のみ）
- 推定は「（推定）」を付ける: `1,200億円（推定）`（自ら算出した場合。§6.2の根拠脚注とセット）
- 「約」と「（推定）」は併用可: `約1,200億円（推定）`

## 10. 禁止事項

以下はすべて**出力前の機械的検証対象**。1件でも該当すればStep 11（品質チェック）で不合格とし、修正後に再チェックする。

| # | 禁止事項 | 判定方法 |
|---|---|---|
| 1 | **HTMLの多用** | 許可タグは `<details>` `<summary>` `<br>` の3種のみ。それ以外のHTMLタグ（`<font>` `<span>` `<div>` `<table>` `<img>` 等）が1つでもあればNG |
| 2 | **画像への依存** | `![](...)` 記法・外部画像URL・badge画像が1つでもあればNG。図はMermaidか表で自己完結させる |
| 3 | **リンク切れ** | 文書内アンカーは§2.3の規則で全件検証。外部URLは調査時にアクセス確認済みのもののみ（§6.1）。未確認URLの記載はNG |
| 4 | **プレースホルダ残し** | `{{` `}}` `TODO` `TBD` `FIXME` `XXX` `[要確認]` `◯◯（仮）` のいずれかが本文に残っていればNG（SKILL.md Step 10 完了判定と同一） |
| 5 | **偽の出典** | 実在確認していないURL・資料名の記載（§6.2。推定表記で代替する） |
| 6 | **禁止Markdown記法** | Setext見出し・インデントコードブロック・行末2スペース改行・`$$`数式（§2.1） |
| 7 | **禁止Mermaid図種** | §3.1の5種以外のMermaidブロック |
| 8 | **見出し規則違反** | §4の正規表現検証に不合格の見出し |
| 9 | **本文絵文字** | 表・リスト外の絵文字（`STYLE_GUIDE.md` §7.2） |
| 10 | **分冊出力** | 1レポートが複数ファイルに分かれている（§1.3） |

### 10.1 出力前セルフ検証手順

Step 10の最後（Step 11の前）に、生成したファイルに対して以下を順に実行する:

1. プレースホルダ検索: `{{`, `TODO`, `TBD`, `FIXME` をファイル内検索し、0件であることを確認
2. 見出し検証: §4の正規表現で全見出しを検証
3. 目次検証: 目次リンクと本文アンカーの突合（§2.3）
4. 脚注検証: 本文の `[^n]` と定義の `[^n]:` が1対1対応、連番であることを確認
5. HTML検証: `<` を検索し、許可3タグ以外がないことを確認
6. Mermaid検証: 各ブロックの図種が許可5種であること、日本語ラベルがクォートされていることを目視確認

すべて通過後、`CHECKLIST.md` の9領域チェック（Step 11）に進む。
