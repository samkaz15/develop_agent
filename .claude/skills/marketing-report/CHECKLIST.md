# CHECKLIST — レポート品質セルフレビュー基準

本ファイルは、`SKILL.md` の Step 11（品質チェック）で使用する必須セルフレビューの判定基準を定義する。9領域すべてを判定文（Pass/Fail が機械的に決められる文）で規定し、「判定可能でなければ基準ではない」の原則に従う。レポートは本チェックの合格ラインを超えるまで出荷（ユーザーへの完了報告）してはならない。

| 項目 | 内容 |
|---|---|
| Version | 1.0.0 |
| Status | Active |
| Last Updated | 2026-07-11 |

---

## 運用ルール

1. **全項目をPass/Failで判定する。** 「ほぼOK」「軽微」などの中間判定は禁止。迷ったらFail。
2. **Fail→修正→再チェック。** Fail項目は修正後、その項目だけでなく**同一領域の全項目**を再チェックする（修正が別項目を壊すことがあるため）。
3. **チェック結果を完了報告に含める。** 末尾の「チェック結果記録テンプレート」を埋め、ユーザーへの完了報告に添付する。省略禁止。
4. **同じFailが2回続いたらルール側を改訂する。** 別レポートで同一項目のFailが2回発生したら、原因となるルールファイル（`TEMPLATE.md` / `STYLE_GUIDE.md` / `WRITING_RULES.md` 等）を改訂し、Versionを上げる（`SKILL.md` §4 改善ループ準拠）。改訂内容はチェック記録の備考欄に残す。
5. **チェックの順序**: ①〜⑨の領域順に行う。上流（情報不足・論理矛盾）のFailは下流の修正を無意味にするため、①②が全Passになるまで③以降へ進まない。
6. **必須項目と任意項目**: 各項目には「必須 / 任意」の区分がある。合格ラインは末尾の「最終出荷判定」に従う。

## ① 情報不足チェック

| # | 区分 | 判定文 |
|---|---|---|
| 1-1 | 必須 | レポート冒頭の「前提条件」に、Step 1の6項目（種別 / 読者 / 意思決定事項 / スコープ / 分量 / 前提資料）がすべて明記されている |
| 1-2 | 必須 | Key Findings が3個以上あり、各Findingに根拠（出典付きの事実または明記された推定）が1つ以上対応している |
| 1-3 | 必須 | 外部データを使った記述すべてに出典と調査日時が付いている（出典なしの外部数値が0件） |
| 1-4 | 必須 | 確認できなかった数値はすべて「推定」または「要計測」と明記されている（無印の不確実数値が0件） |
| 1-5 | 任意 | 読者の意思決定に必要な情報（判断材料・選択肢・リスク）のうち、欠けているものがあれば「前提条件」または「Risks」に欠落として明記されている |
| 1-6 | 任意 | 出典の信頼度Tier（`RESEARCH_METHOD.md` 準拠）が主要な根拠に付記されている |

## ② 論理矛盾チェック

| # | 区分 | 判定文 |
|---|---|---|
| 2-1 | 必須 | Executive Summary の結論と、本文各章の結論に矛盾が0件である |
| 2-2 | 必須 | すべての推奨（Strategic Recommendations）が、本文中のKey FindingsまたはInsightsから追跡できる（本文に根拠のない推奨が0件） |
| 2-3 | 必須 | 同一の数値（市場規模・KPI現状値等）が複数章に登場する場合、すべての章で値と単位が一致している |
| 2-4 | 必須 | 各Insightが「事実→解釈→So What」の3段で構成され、So Whatテスト（この発見は読者の意思決定を変えうるか？に Yes と答えられる）に合格している |
| 2-5 | 任意 | 「AだからB」の因果記述について、相関にすぎないものが因果と断定されていない（断定表現には根拠または「推定」明記がある） |
| 2-6 | 任意 | 反証・不利な事実が存在する場合、無視されずにRisksまたは本文で扱われている |

## ③ 重複チェック

| # | 区分 | 判定文 |
|---|---|---|
| 3-1 | 必須 | 同一の内容（同じ主張＋同じ根拠）を述べる段落が複数章に存在しない（Executive Summaryによる要約再掲は除く） |
| 3-2 | 必須 | 同一の表・図が2回以上掲載されていない |
| 3-3 | 任意 | 類似する2つの章（例: 戦略と施策）で役割分担が明確で、どちらに書くべきか迷う記述が残っていない |
| 3-4 | 任意 | 同じ用語に対する定義・説明が2回以上繰り返されていない（初出時に1回定義し、以降は参照する） |

## ④ 読みづらさチェック

| # | 区分 | 判定文 |
|---|---|---|
| 4-1 | 必須 | 1文が60字以内である（超過する文が全体の5%未満。60字超の文は分割する） |
| 4-2 | 必須 | 1段落が3文以内である（超過する段落が0件。長い段落は箇条書きか分割で解消する） |
| 4-3 | 必須 | 目次があり、目次の項目と本文の見出しが完全に一致している |
| 4-4 | 必須 | 各章の見出しがメッセージ化されている（名詞止めの見出しが主要章に0件。例:「市場分析」ではなく「市場は年率X%で拡大し、参入余地がある」） |
| 4-5 | 任意 | 見出し階層が3レベル以内（#, ##, ###）に収まっている |
| 4-6 | 任意 | 専門用語・略語は初出時に補足説明が付いている（無説明の専門用語が0件） |
| 4-7 | 任意 | 各章の第1文が結論になっている（`SKILL.md` 品質基準1: 結論ファースト） |

## ⑤ 図表不足チェック

| # | 区分 | 判定文 |
|---|---|---|
| 5-1 | 必須 | 主要な主張（Key Findings・推奨・KPI・ロードマップ）の8割以上が表・図・リストのいずれかで裏付けられている |
| 5-2 | 必須 | すべての表が規定内である（列数7以内・行数15以内。超える場合は分割または付録送りにする） |
| 5-3 | 必須 | すべてのMermaid図がレンダリング可能である（構文エラー0件。ノード数12以内） |
| 5-4 | 必須 | すべての図表に番号とタイトル（および外部データの場合は出典）が付いている |
| 5-5 | 任意 | 数値比較は文章ではなく表になっている（3個以上の数値を並べる文が0件） |
| 5-6 | 任意 | 図表が対応する本文の直後に配置され、本文から参照されている（参照されない図表が0件） |

## ⑥ 根拠不足チェック

| # | 区分 | 判定文 |
|---|---|---|
| 6-1 | 必須 | 推定値・仮定値がすべて「推定」と明記され、算出根拠または仮定の内容が書かれている |
| 6-2 | 必須 | 重要な意思決定に関わる根拠が `RESEARCH_METHOD.md` のTier 3（業界レポート）以上の情報源に基づいている（Tier 4以下のみに依存する主要結論が0件） |
| 6-3 | 必須 | 割合・比率を述べる箇所すべてに分母が明示されている（分母不明の「X%」「多くの」が0件） |
| 6-4 | 必須 | 存在しない出典・URL・数値の捏造が0件である（全出典が実在確認済み、または「推定」明記） |
| 6-5 | 任意 | サンプルの偏り・データの鮮度など、根拠の限界がRisksに記載されている |

## ⑦ KPI不足チェック

| # | 区分 | 判定文 |
|---|---|---|
| 7-1 | 必須 | すべてのKPIに「定義式 / 目標値 / 計測方法」の3点セットが揃っている（1点でも欠けるKPIが0件） |
| 7-2 | 必須 | KPIの現状値が記載されている（不明の場合は「要計測」と明記され、計測手順がNext Actionsにある） |
| 7-3 | 必須 | KPIツリー（最上位目標→先行指標→施策別指標）が1枚の図または表で表現されている |
| 7-4 | 任意 | 各KPIに確認頻度（日次/週次/月次）と確認担当の想定が付いている |
| 7-5 | 任意 | 施策とKPIが対応している（どのKPIも動かさない施策が0件、どの施策にも紐づかないKPIが0件） |

## ⑧ 改善提案不足チェック

| # | 区分 | 判定文 |
|---|---|---|
| 8-1 | 必須 | 施策が5個以上列挙されている |
| 8-2 | 必須 | 全施策にICEまたはRICEスコアが付き、優先順位が確定している（採点基準は `ANALYSIS_FRAMEWORK.md` 準拠） |
| 8-3 | 必須 | 推奨案が1つに確定している（「AもBもあり得る」で終わる両論併記が0件） |
| 8-4 | 必須 | 推奨しない代替案が比較表に残っており、推奨理由が評価軸（インパクト/実現性/リスク/投資額）で説明されている |
| 8-5 | 任意 | 各施策に期待効果・担当想定・工数規模（S/M/L）が付いている |
| 8-6 | 任意 | 改善仮説が「もし〜すれば、〜になるはずだ」形式で3個以上あり、確信度と検証方法が付いている |

## ⑨ 実行可能性チェック

| # | 区分 | 判定文 |
|---|---|---|
| 9-1 | 必須 | Next Actions のすべての項目が「誰が・何を・いつまでに」の3要素を持つ（1要素でも欠ける項目が0件） |
| 9-2 | 必須 | ロードマップのフェーズ移行判定条件（Go/No-Go基準）がすべて数値で書かれている（「手応えがあれば」等の定性条件が0件） |
| 9-3 | 必須 | 全施策がロードマップ上に配置されている（宙に浮いた施策が0件） |
| 9-4 | 任意 | 施策間の依存関係が明示されている（先行施策の完了を要する施策に依存の記載がある） |
| 9-5 | 任意 | 最初の1週間で着手できるアクションが1つ以上含まれている |

## 読者別の追加チェック

レポートの読者区分（前提条件に記載）に応じて、以下を追加で判定する。該当読者の追加チェックは**すべて必須扱い**とする。

### 経営層向け

| # | 判定文 |
|---|---|
| E-1 | Executive Summary が1ページ以内（Markdown換算で見出し含め40行以内）に収まっている |
| E-2 | Executive Summary だけを読んで意思決定できる（結論・根拠となる数字・推奨・求める判断の4要素がすべて含まれる） |
| E-3 | 求める意思決定が明示されている（「ご判断ください」ではなく「XをYの条件で承認いただきたい」の粒度） |
| E-4 | 投資額・リソースの規模感が数値で書かれている（「一定の投資」等の曖昧表現が0件） |

### 投資家向け

| # | 判定文 |
|---|---|
| I-1 | TAM / SAM / SOM が算出ロジック付きで記載されている |
| I-2 | ユニットエコノミクス（LTV / CAC / LTV:CAC比 / 回収期間のうち算出可能なもの）が記載され、不明値は「要計測」と明記されている |
| I-3 | リスクが3個以上開示され、各リスクに発生可能性と対応方針が付いている（リスク項目ゼロのレポートは投資家向けとしてFail） |
| I-4 | 成長性が数値（CAGR・成長ドライバー）で語られ、希望的観測のみの成長記述が0件である |
| I-5 | 競合が明記されている（「競合はいない」という記述をする場合、代替手段の分析が伴っている） |

## 最終出荷判定

以下の**すべて**を満たした場合のみ「出荷可」とする。1つでも満たさなければ修正して再チェックする。

| 条件 | 合格ライン |
|---|---|
| 必須項目（①〜⑨の「必須」区分） | **全項目Pass（100%）** |
| 任意項目（①〜⑨の「任意」区分） | **8割以上Pass**（Failした任意項目は理由を記録に残す） |
| 読者別追加チェック | 該当読者の項目 **全Pass** |
| `SKILL.md` §2 品質基準9項目 | **全項目Pass**（本チェックリストと重複する項目は本結果を流用してよい） |

- 3回連続で最終出荷判定に到達しない場合は、修正の繰り返しをやめて構成レベル（Step 3〜6）からやり直す。
- 任意項目のFailを「今回は許容」とした場合、次回同種レポートで同じ項目がFailしたら運用ルール4（ルール改訂）を発動する。

## チェック結果記録テンプレート

完了報告には以下の表を埋めて添付する。

```markdown
### 品質チェック結果（CHECKLIST.md v1.0.0）

| 領域 | 必須 Pass/総数 | 任意 Pass/総数 | Fail項目と修正内容 |
|---|---|---|---|
| ① 情報不足 | /4 | /2 | |
| ② 論理矛盾 | /4 | /2 | |
| ③ 重複 | /2 | /2 | |
| ④ 読みづらさ | /4 | /3 | |
| ⑤ 図表不足 | /4 | /2 | |
| ⑥ 根拠不足 | /4 | /1 | |
| ⑦ KPI不足 | /3 | /2 | |
| ⑧ 改善提案不足 | /4 | /2 | |
| ⑨ 実行可能性 | /3 | /2 | |
| 読者別追加（対象: ）| / | — | |

- 必須項目: XX/32 Pass（100%必達）
- 任意項目: XX/18 Pass（80%以上で合格: 15/18以上）
- 最終判定: 出荷可 / 出荷不可
- 特記事項（2回目のFailによるルール改訂の有無、許容した任意Failの理由）:
```

記録時の規則:

1. Fail→修正した項目は「Fail項目と修正内容」欄に「項目番号: 修正内容の要約（1行）」で残す。修正後の最終状態がPassでも、Failした履歴は消さない（改善ループの入力になるため）。
2. 同一項目のFailが過去レポートと合わせて2回目になった場合、特記事項に「ルール改訂: {対象ファイル} {改訂内容}」を記載する。
3. この記録はユーザーへの完了報告に含める（`SKILL.md` Step 11 完了判定の一部）。
