# PROMPTS — 分析タイプ別 再利用プロンプト集

本ファイルは、Marketing Report Skill（`SKILL.md`）で扱う16種の分析タイプごとに、**コピペしてそのまま使える再利用プロンプト**を提供する。すべてのプロンプトは `SKILL.md` の11ステップフローと `TEMPLATE.md` の章構成に沿った出力を強制するように書かれており、誰が・どのプロンプトを使っても同一品質のレポートが生成されることを目的とする。

| Version | 1.0.0 |
| Status | Active |
| Last Updated | 2026-07-11 |

## プロンプトの使い方

### 変数の埋め方

- 各プロンプト本文中の `{{変数名}}` を、実行前にすべて具体値へ置換する。**未置換の `{{ }}` が1つでも残った状態で実行してはならない**（判定: 実行前に `{{` を検索して0件であること）。
- 値が不明な変数は削除せず、`{{変数名}}` の位置に「不明（調査して補完せよ）」と書く。プロンプト実行側（Claude）はStep 1の完了判定に従い、推測せず質問するか、前提として明記する。
- 日付・期間・地域は必ず明示する（例: 「直近」ではなく「2025-07〜2026-06」）。曖昧な期間指定はStep 2の出典管理を壊す。

### 共通変数（全プロンプトで使用）

| 変数 | 内容 | 例 |
|---|---|---|
| `{{読者}}` | 経営層 / 投資家 / 実務者 / 社内共有 のいずれか | 経営層 |
| `{{意思決定}}` | このレポートで読者に決めてほしいこと1文 | 新市場参入のGo/No-Go |
| `{{対象}}` | 分析対象（自社/製品/市場/競合名） | 自社アプリ「{{製品名}}」 |
| `{{期間}}` | 分析対象期間（開始〜終了を明記） | 2025-07-01〜2026-06-30 |
| `{{地域}}` | 対象地域 | 日本国内 |
| `{{分量}}` | 希望分量（ページ/字数目安） | A4換算10ページ以内 |

### 複数プロンプトの組み合わせ方

1. **直列（推奨）**: 「市場調査 → 競合分析 → 価格分析」のように上流から下流へ順に実行し、前のレポートのKey Findingsを次のプロンプトの `{{前提資料}}` に渡す。1本の統合レポートにする場合、最後に統合プロンプト（各レポートのExecutive Summaryを入力に、`TEMPLATE.md` 構成で再編）を実行する。
2. **並列**: 独立した分析（例: TikTok / Instagram / YouTube）を並列実行し、「SNS分析（総合）」プロンプトで統合する。並列時は `{{期間}}` と `{{対象}}` を全プロンプトで完全一致させる（判定: 3変数の値が全プロンプト間で文字列一致）。
3. **禁止**: 1回の実行に3種類以上の分析タイプを混載すること。分析の完了判定（`SKILL.md` Step 4: インサイト3〜7個）が薄まるため、分割して実行する。

### カスタムプロンプトの追加規則

- 本ファイル末尾の「新規プロンプト追加テンプレート」を使い、①〜⑤の5要素を欠落なく埋める（5要素が揃わないプロンプトは追加不可）。
- プロンプト本文には必ず「`SKILL.md` の11ステップに従うこと」「`TEMPLATE.md` の章構成で出力すること」の2指示を含める。
- 追加後、`SKILL.md` §0 の表のプロンプト数（現在16種）を更新し、Versionをマイナーバージョンアップする（例: 1.0.0 → 1.1.0）。

## 1. 市場調査（Market Research）

**① 用途（いつ使う）**: 新市場への参入判断、既存市場の再評価、事業計画の市場規模章の作成時。TAM/SAM/SOMと成長性の定量把握が目的のとき。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{市場名}}` `{{地域}}` `{{期間}}` `{{分量}}` `{{自社の参入形態}}`（新規参入/既存拡大/未定）

**③ プロンプト本文**

```text
あなたはトップティアコンサルティングファームのシニアアナリストとして、
「{{市場名}}」（対象地域: {{地域}}、対象期間: {{期間}}）の市場調査レポートを作成せよ。
読者は{{読者}}、意思決定事項は「{{意思決定}}」、分量は{{分量}}。
自社の参入形態は「{{自社の参入形態}}」である。

必ず SKILL.md の11ステップフロー（目的確認→調査→情報整理→インサイト抽出→仮説立案→
戦略立案→改善案作成→KPI整理→ロードマップ作成→レポート生成→品質チェック）を省略せずに実行し、
最終出力は TEMPLATE.md の章構成に従うこと。

分析には以下を必ず含めること:
1. 市場規模: TAM/SAM/SOMの3層を算出し、各層の算出ロジック（トップダウン/ボトムアップ）と
   計算式を明記する。確認できない数値は「推定」と明記する。
2. 成長性: 過去3年のCAGRと今後3年の成長予測。予測の根拠となる成長ドライバーを3つ以上特定する。
3. 市場構造: バリューチェーン、主要プレイヤーのシェア（上位5社目安）、参入障壁の高さ評価。
4. 顧客セグメント: 主要セグメント3〜5個と、各セグメントの規模・ニーズ・支払意思。
5. PEST分析: 政治/経済/社会/技術の変化のうち、市場に影響する要因を各1つ以上。
6. 参入した場合のGo/No-Go判断材料: 参入すべき条件と撤退基準を数値で提示する。
全事実に出典と調査日時を付け、Key Findings候補を10件以上収集してから執筆すること。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠のフルレポート1本。TAM/SAM/SOMは算出式付きの表、市場構造はMermaid図、Go/No-Go判断材料はStrategic Recommendationsの章に数値条件で記載。

**⑤ 品質の落とし穴**
- TAM/SAM/SOMを出典なしの一発数値で書く（算出式と一次データの出典がなければFail）。
- 「成長市場である」で終わる（誰にとっての機会かのSo Whatがないと`CHECKLIST.md`②で落ちる）。
- 古い市場規模データの基準年を明記しない（鮮度リスクをRisksに書く）。

## 2. 競合分析（Competitive Analysis）

**① 用途（いつ使う）**: 競合の戦略・強み弱みを把握して自社の差別化ポイントを決めるとき。価格改定・機能追加・ポジショニング変更の前。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{自社名}}` `{{競合リスト}}`（3〜5社推奨）`{{比較したい軸}}` `{{期間}}` `{{地域}}`

**③ プロンプト本文**

```text
あなたは競合インテリジェンスの専門家として、{{自社名}}の競合分析レポートを作成せよ。
対象競合: {{競合リスト}}。比較軸の希望: {{比較したい軸}}。
対象期間: {{期間}}、地域: {{地域}}、読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. 競合比較表: 各社について 価格 / 主要機能 / ターゲット顧客 / チャネル / 直近の動き
   （資金調達・新機能・提携）を1枚の表に整理する。セルに出典番号を付ける。
2. ポジショニングマップ: 顧客が購買を決める2軸を選定理由付きで選び、2x2で各社を配置する。
3. 各社のSWOTの要点（各社3行以内）と、自社視点での「攻めどころ」「守りどころ」。
4. 4P比較（Product/Price/Place/Promotion）で自社と上位競合2社の差分を明記する。
5. 競合の次の一手の予測: 各社1つ、根拠となる事実（採用動向・IR・リリース等）付きで。
6. 差別化戦略の推奨案1つ: 代替案2つとの比較表（インパクト/実現性/リスク/投資額）付きで確定する。
公開情報のみを使い、憶測は「推定」と明記すること。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。競合比較表（列は7列以内、超える場合は表を分割）、ポジショニングマップはMermaid quadrantChart、推奨案は1つに確定。

**⑤ 品質の落とし穴**
- 競合の機能一覧を並べるだけで「自社はどうすべきか」がない（Step 4のSo What欠落）。
- ポジショニングマップの軸選定理由がない（軸は顧客の購買決定要因から選ぶ）。
- 競合の非公開数値（売上等）を断定で書く（推定明記がなければ`CHECKLIST.md`⑥でFail）。

## 3. SEO分析（SEO Analysis）

**① 用途（いつ使う）**: オーガニック流入の停滞・下落の原因究明、コンテンツ戦略の立て直し、SEO投資の優先順位決定時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{対象サイトURL}}` `{{主要キーワード群}}` `{{期間}}` `{{手元データ}}`（Search Console/GA4のエクスポート等、なければ「なし」）

**③ プロンプト本文**

```text
あなたはSEOストラテジストとして、{{対象サイトURL}}のSEO分析レポートを作成せよ。
主要キーワード群: {{主要キーワード群}}。対象期間: {{期間}}。
手元データ: {{手元データ}}。読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. 現状診断: オーガニック流入の推移（月次）、上位流入キーワードと順位、CTRの分布。
   手元データがない項目は「要計測」と明記し、計測手順をNext Actionsに含める。
2. キーワード分析: 検索意図（Know/Do/Buy/Go）別にキーワードを分類し、
   「順位x検索意図」のマトリクスで機会領域を特定する。
3. 競合ギャップ: 上位表示サイト2〜3個と自社のコンテンツ差分（網羅性/E-E-A-T要素/構造化データ）。
4. テクニカル要因: Core Web Vitals、インデックス状況、内部リンク構造の課題を優先度付きで列挙。
5. コンテンツ改善の優先順位: 「リライト/新規作成/統合/削除」の4分類で対象ページを整理し、
   ICEスコアで順位付けする。
6. 改善仮説を3つ以上、「もし〜すれば流入が〜%改善するはず」形式で、検証方法付きで提示する。
アルゴリズム変動など不確実な要因はRisksに記載すること。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。キーワードマトリクスは表、改善施策はICEスコア付き優先順位表、KPIは「オーガニックセッション数」等の定義式・目標値・計測方法の3点セット。

**⑤ 品質の落とし穴**
- 順位データの取得日を書かない（順位は日次変動するため調査日時必須）。
- 「コンテンツを充実させる」という施策粒度（誰が・どのページを・いつまでにが必要）。
- 手元データなしで流入数値を捏造する（不明は「要計測」、これは`SKILL.md` §3のエラー処理準拠）。

## 4. SNS分析（総合 / Social Media Overview）

**① 用途（いつ使う）**: 複数SNSチャネルの横断評価、SNS予算・リソース配分の見直し、チャネルポートフォリオ戦略の策定時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{対象アカウント群}}`（チャネル名とアカウント名）`{{期間}}` `{{目的KPI}}`（認知/流入/CV等）`{{手元データ}}`

**③ プロンプト本文**

```text
あなたはソーシャルメディアストラテジストとして、{{対象アカウント群}}の
SNS総合分析レポートを作成せよ。対象期間: {{期間}}、SNS運用の目的KPI: {{目的KPI}}。
手元データ: {{手元データ}}。読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. チャネル横断サマリー表: チャネルごとに フォロワー数 / 増減率 / 平均エンゲージメント率 /
   投稿頻度 / 主要KPIへの貢献 を1枚に整理する（データ不明のセルは「要計測」）。
2. チャネル別の役割定義: 各チャネルをファネル（認知/興味/比較/CV/ファン化）のどこに
   位置づけるかを明示し、役割が重複・欠落しているチャネルを特定する。
3. コンテンツタイプ分析: 投稿を3〜6タイプに分類し、タイプ別のエンゲージメント率を比較する。
4. 競合ベンチマーク: 同業アカウント2〜3個との投稿頻度・エンゲージメント率比較。
5. リソース配分の推奨: 「強化/維持/縮小/撤退」の4象限に各チャネルを配置し、
   推奨配分を1案に確定する（代替案との比較表付き）。
6. 改善仮説3つ以上（確信度・検証方法付き）と、90日ロードマップ。
エンゲージメント率の定義式（分子・分母）を必ず明記すること。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。チャネル横断表＋ファネル配置のMermaid図＋強化/維持/縮小/撤退のマトリクス。KPIツリーは目的KPIを頂点にチャネル別指標へ分解。

**⑤ 品質の落とし穴**
- エンゲージメント率の分母がチャネル間で不統一（フォロワー基準かリーチ基準かを明記）。
- 全チャネル「強化」の総花的提案（リソース有限の前提で縮小・撤退候補を必ず出す）。
- フォロワー数の多寡だけで評価する（目的KPIへの貢献で評価しないと意思決定に使えない）。

## 5. TikTok分析

**① 用途（いつ使う）**: TikTokアカウントの成長停滞の診断、ショート動画の勝ちパターン特定、TikTok参入判断時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{アカウント名}}` `{{期間}}` `{{手元データ}}` `{{ベンチマーク対象}}`（同業アカウント2〜3個）

**③ プロンプト本文**

```text
あなたはショート動画グロースの専門家として、TikTokアカウント「{{アカウント名}}」の
分析レポートを作成せよ。対象期間: {{期間}}、ベンチマーク対象: {{ベンチマーク対象}}。
手元データ: {{手元データ}}。読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. 基礎指標の推移: 再生数 / 視聴完了率 / いいね率 / コメント率 / シェア率 / 保存率 /
   フォロワー転換率 を月次で整理する（不明は「要計測」）。
2. 動画の分解分析: 上位動画と下位動画それぞれ5本を、フック（冒頭2秒）/ 尺 / 構成 /
   ハッシュタグ / 投稿時間帯 の5要素で比較し、勝ちパターン仮説を導く。
3. 視聴維持カーブの型: 「冒頭離脱型 / 中だるみ型 / 完走型」のどれに該当するかを判定し、
   型ごとの改善打ち手を対応させる。
4. アルゴリズム適合度: 投稿頻度・トレンド音源活用・コメント返信速度の3点を
   ベンチマーク対象と比較する。
5. 改善仮説を3つ以上（例: 「フックを疑問形にすれば視聴完了率が+X ptになるはず」）、
   確信度と検証方法（次のN本でA/B）付きで提示する。
6. 90日の投稿実験ロードマップ（週次の本数・タイプ・判定条件を数値で）。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。上位/下位動画の比較表、視聴維持カーブの型判定、実験ロードマップは週次表。KPIは視聴完了率とフォロワー転換率を必ず含む。

**⑤ 品質の落とし穴**
- 再生数だけで良し悪しを判定する（完了率・保存率などの質的指標を併記しないとFail）。
- 「バズった動画を増やす」という再現不能な提案（フック・構成レベルまで分解する）。
- アルゴリズム仕様を断定で書く（公式発表以外は「推定」と明記）。

## 6. Instagram分析

**① 用途（いつ使う）**: Instagramアカウントのエンゲージメント低下診断、フィード/リール/ストーリーズの配分見直し、UGC・ショッピング活用の判断時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{アカウント名}}` `{{期間}}` `{{手元データ}}` `{{ベンチマーク対象}}`

**③ プロンプト本文**

```text
あなたはInstagramマーケティングの専門家として、アカウント「{{アカウント名}}」の
分析レポートを作成せよ。対象期間: {{期間}}、ベンチマーク対象: {{ベンチマーク対象}}。
手元データ: {{手元データ}}。読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. フォーマット別分析: フィード / リール / ストーリーズ / ライブ の4フォーマットについて、
   リーチ / エンゲージメント率 / 保存率 / プロフィール遷移率 を比較表にする。
2. リーチ内訳: フォロワー内リーチとフォロワー外リーチ（発見タブ・リール経由）の比率を分析し、
   新規獲得とファン維持のどちらに課題があるかを判定する。
3. 投稿分析: 上位・下位投稿各5件を ビジュアル要素 / キャプション構成 / ハッシュタグ /
   投稿時間 で比較し、勝ちパターン仮説を導く。
4. プロフィール導線: プロフィール閲覧→フォロー転換率、リンククリック率を評価する。
5. コミュニティ健全度: コメントの質（質問/称賛/苦情の比率）、返信率・返信速度。
6. フォーマット配分の推奨1案（現状配分との比較表付き）、改善仮説3つ以上
   （確信度・検証方法付き）、90日ロードマップ（判定条件は数値）。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。フォーマット比較表を中心に、リーチ内訳の図、推奨配分は「現状→推奨」の変化が1枚で分かる表。

**⑤ 品質の落とし穴**
- フォロワー数の増減だけを見る（リーチ内訳と保存率を見ないと打ち手を誤る）。
- ハッシュタグ数の議論に終始する（現在の到達構造ではビジュアルとフックが支配的、根拠付きで論じる）。
- 「世界観を統一する」という判定不能な提案（色数・トーン・投稿比率など数値で規定する）。

## 7. YouTube分析

## 13. LTV分析（LTV Analysis）

**① 用途（いつ使う）**: 顧客生涯価値の算出と改善レバーの特定、マーケ投資上限の設定、プラン・セグメント別の収益性評価時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{対象事業}}` `{{ビジネスモデル}}`（サブスク/EC/その他）`{{期間}}` `{{手元データ}}`（ARPU・解約率・粗利率・購買履歴等）

**③ プロンプト本文**

```text
あなたはユニットエコノミクスの専門家として、{{対象事業}}（モデル: {{ビジネスモデル}}）の
LTV分析レポートを作成せよ。対象期間: {{期間}}、手元データ: {{手元データ}}。
読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. LTV定義の確定: 採用する定義式を明記する（例: サブスクなら LTV = ARPU x 粗利率 ÷ 解約率、
   ECなら 平均購入単価 x 購入頻度 x 継続期間 x 粗利率）。粗利ベースか売上ベースかを必ず宣言する。
2. 現状LTVの算出: 全体と、セグメント別（プラン / 獲得チャネル / 顧客属性のうち
   データが取れる軸）のLTVを算出する。入力値が推定の場合は推定と明記する。
3. 感度分析: 解約率 / ARPU / 粗利率 をそれぞれ±10%動かしたときのLTV変化を表にし、
   最もレバレッジの効く変数を1つ特定する。
4. LTV:CAC比の評価: CACデータがあれば比率を算出し、3:1を参考基準として健全性を判定する
   （基準値は業界により幅がある旨を明記）。回収期間（CAC Payback）も算出する。
5. LTV改善施策5個以上（解約低減/単価向上/頻度向上/粗利改善に分類）をICE/RICEで順位付けし、
   推奨着手順を確定する。
6. KPIツリー（頂点: LTV、分解: 定義式の各変数）と、四半期ロードマップ（判定条件は数値）。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。定義式→算出表→感度分析表→施策優先順位の流れ。KPIツリーはLTVの定義式と完全に対応。

**⑤ 品質の落とし穴**
- 定義式を書かずにLTVの数値だけ出す（売上/粗利ベースの不一致は経営判断を誤らせる）。
- 平均LTVだけでセグメント差を見ない（平均は分布の歪みを隠す）。
- 解約率が期間で変動するのに単一値で永久係数計算する（コホート別解約率の確認をRisksに書く）。

## 14. CAC分析（CAC Analysis）

**① 用途（いつ使う）**: 顧客獲得コストの実態把握、チャネル別の投資効率評価、マーケ予算の再配分判断時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{対象事業}}` `{{期間}}` `{{手元データ}}`（チャネル別費用・獲得数・人件費配賦の有無）`{{LTV値}}`（あれば）

**③ プロンプト本文**

```text
あなたはマーケティングファイナンスの専門家として、{{対象事業}}のCAC分析レポートを作成せよ。
対象期間: {{期間}}、手元データ: {{手元データ}}、参照LTV: {{LTV値}}。
読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. CAC定義の確定: 分子（広告費のみか、人件費・ツール費・代理店費込みか）と
   分母（新規顧客の定義、無料登録か有料転換か）を明記し、Blended CACと
   Paid CACを区別して算出する。
2. チャネル別CAC: 各獲得チャネルの 費用 / 獲得数 / CAC / 獲得数構成比 を表にし、
   CACの高低だけでなく獲得ボリュームとの2軸で評価する。
3. トレンド分析: CACの月次推移と、悪化/改善の要因分解（CPM高騰・CVR変化・ミックス変化）。
4. LTV:CAC比とPayback期間のチャネル別評価（LTVが不明なら「要計測」とし算出をNext Actionsへ）。
5. 予算再配分の推奨1案: 「増額/維持/減額/停止」をチャネル別に確定し、
   現状配分との比較表と、再配分後の獲得数・CACの試算（前提明記の推定）を付ける。
6. 限界CACの考察: 予算を増やしたときのCAC上昇（逓減）の兆候があるかを評価し、
   増額上限の目安を数値で提示する。改善仮説3つ以上と検証方法も含める。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。定義宣言→チャネル別表→再配分比較表。推奨は1案に確定し、試算の前提はすべて脚注化。

**⑤ 品質の落とし穴**
- 分子・分母の定義を書かない（CACは定義次第で2倍以上変わる。定義なしはFail）。
- オーガニック獲得をPaid CACに混ぜてCACを過小評価する（Blended/Paidを分ける）。
- 平均CACで判断する（限界CACの逓減を無視した増額提案は危険、Risksに記載）。

## 15. 継続率（リテンション）分析

**① 用途（いつ使う）**: 解約・離脱の構造把握、プロダクトのPMF評価、オンボーディング/復帰施策の設計時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{対象製品}}` `{{継続の定義}}`（例: 7日以内の再ログイン）`{{期間}}` `{{手元データ}}`（コホート別利用ログ等）

**③ プロンプト本文**

```text
あなたはリテンショングロースの専門家として、{{対象製品}}の継続率分析レポートを作成せよ。
継続の定義: {{継続の定義}}。対象期間: {{期間}}、手元データ: {{手元データ}}。
読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること。

分析には以下を必ず含めること:
1. コホートリテンション: 登録月別コホートで D1 / D7 / D30（サブスクなら M1 / M3 / M6）の
   継続率をコホート表にする。継続の定義（分子・分母）を冒頭で宣言する。
2. カーブの型判定: リテンションカーブが「フラット化型（一定水準で下げ止まる）/
   減衰継続型（下げ止まらない）/ スマイル型（復帰で上昇）」のどれかを判定し、
   フラット化する水準と時期を数値で示す。下げ止まらない場合はPMF課題として明記する。
3. セグメント分解: 曜日・獲得チャネル・ペルソナ（または利用目的）別に継続率を分解し、
   高継続セグメントと低継続セグメントの行動差分を特定する。
4. アクティベーション分析: 高継続ユーザーが初期に行う行動（アハ・モーメント候補）を
   2〜3個特定し、「初回N日以内に行動Xをしたユーザーは継続率が+Y pt」の形式で示す。
5. 業界ベンチマーク比較: 同カテゴリのD1/D7/D30目安と比較する（出典必須、なければ推定と明記）。
6. 改善仮説を3つ以上（オンボーディング/習慣化/復帰の3レイヤーから最低1つずつ）、
   確信度・検証方法付きで提示し、KPIツリーと90日ロードマップ（判定条件は数値）を作る。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠。コホート表＋カーブ型判定＋セグメント分解表＋アクティベーション条件の定量記述。改善仮説はレイヤー別に3つ以上。

**⑤ 品質の落とし穴**
- 継続の定義（分子・分母・判定ウィンドウ）を書かない（定義が変わると数値は別物になる）。
- 平均継続率だけ見てコホート・セグメント差を見ない（改善レバーが特定できない）。
- 相関を因果と断定する（「行動Xで継続率+Y pt」は相関であり、検証はA/Bで行うと明記）。

## 16. A/Bテスト分析

**① 用途（いつ使う）**: 実施済みA/Bテストの結果評価と意思決定（採用/棄却/再テスト）、テスト設計のレビュー、テストプログラム全体の振り返り時。

**② 必要な入力**: `{{読者}}` `{{意思決定}}` `{{テスト概要}}`（仮説・変更内容）`{{テストデータ}}`（各群のサンプルサイズ・CV数・期間）`{{主要指標}}` `{{ガードレール指標}}`

**③ プロンプト本文**

```text
あなたは実験計画とグロース統計の専門家として、A/Bテスト分析レポートを作成せよ。
テスト概要: {{テスト概要}}。テストデータ: {{テストデータ}}。
主要指標: {{主要指標}}、ガードレール指標: {{ガードレール指標}}。
読者: {{読者}}、意思決定事項: 「{{意思決定}}」。

SKILL.md の11ステップを省略せず実行し、TEMPLATE.md の章構成で出力すること
（実験レポートに不要な章は統合し、省略理由を前提条件に明記してよい）。

分析には以下を必ず含めること:
1. テスト設計の妥当性評価: 仮説の形式（もし〜なら〜のはず）、サンプルサイズの十分性
   （検出したい効果量に対する検定力80%基準）、期間（最低1週間・曜日周期を跨ぐか）、
   SRM（サンプル比率の乖離）チェックの4点をPass/Failで判定する。
2. 結果の統計評価: 各群のCVRと差分、95%信頼区間、p値（または事後確率）を表にする。
   統計的有意性とビジネス的有意性（効果量が投資判断を変えるか）を分けて評価する。
3. セグメント別の効果差: 新規/既存、デバイス別等での効果の異質性を確認する
   （多重比較の危険性に言及し、セグメント結果は仮説生成用と位置づける）。
4. ガードレール確認: 主要指標の改善が他指標（解約率・客単価・パフォーマンス等）を
   毀損していないかを判定する。
5. 意思決定の推奨: 「採用 / 棄却 / 再テスト」から1つに確定し、判断根拠を明記する。
   再テストの場合は必要サンプルサイズと期間を算出する。
6. 学びの一般化: このテストから得た仮説を3つ以上、次のテスト候補として
   ICEスコア付きで提示する。
```

**④ 出力の期待形式**: `TEMPLATE.md` 準拠（章統合可）。設計妥当性のPass/Fail表→結果統計表→採用/棄却/再テストの1択確定→次のテスト候補リスト。

**⑤ 品質の落とし穴**
- 有意差なし＝効果なしと断定する（検出力不足の可能性を必ず評価する）。
- テスト途中の覗き見（peeking）で有意になった時点で打ち切った結果をそのまま採用する。
- セグメント別の後付け有意差を主結論にする（多重比較の補正なしの深掘りは仮説生成まで）。
