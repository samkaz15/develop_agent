# RESEARCH_METHOD.md — 調査の標準手法

本書は Marketing Report Skill の **Step 2（調査）** で用いる標準手法を定義する。目的は「誰が調査しても、同じ信頼度・同じ透明性のエビデンスセットが揃う」ことであり、Step 2 の完了判定「**Key Findings 候補10件以上・全件に出典タグ付き**」（`SKILL.md` 準拠）を確実に満たすための実務手順・判定基準・テンプレートを提供する。Step 3（情報整理）で使う「事実/解釈/推定」ラベルの付与規則もここで定める。

## 1. 調査設計（Research Design）

### 1.1 原則: 問いのない調査は開始しない

- 調査開始前に**リサーチクエスチョン（RQ）を3〜7個**文章で書き出す。RQが1つも書けない場合は Step 1（目的確認）に戻る。
- RQは「Yes/Noまたは数値で答えられる形」にする。答えの形が定義できない問いはRQとして不合格。

### 1.2 目的→問い→必要データの逆算表

調査は必ず以下の逆算表を作ってから始める。**この表がないまま WebSearch を実行してはならない。**

| 列 | 記入内容 | 記入例 |
|---|---|---|
| 意思決定 | このレポートで読者に決めてほしいこと | 「国内○○市場に2027年参入するか否か」 |
| リサーチクエスチョン（RQ） | 意思決定に必要な問い | RQ1: 市場規模は参入基準（TAM 500億円以上）を満たすか |
| 必要データ | RQに答えるための具体データ | 国内○○市場の2023-2026年市場規模とCAGR |
| 想定情報源 | どのTier（§2）で入手できそうか | Tier 1: 経産省統計 / Tier 2: 調査会社レポート |
| 優先度 | High / Mid / Low | High（意思決定に直結） |

**運用ルール（判定文）**:

- R-01: RQのうち優先度Highが1つもない場合、逆算表は不合格。作り直す。
- R-02: 「必要データ」列が「〜について調べる」のような動詞で終わっている行は不合格。名詞（データ項目名+期間+単位）で書く。
- R-03: 調査時間の50%以上を優先度HighのRQに割く。Low のRQは High が充足するまで着手しない。

### 1.3 調査スコープ定義

Step 1 で確定した内容を、調査開始前に以下の4軸で明文化する（レポートの「前提条件」章にそのまま転記する）。

| 軸 | 定義項目 | 不備の判定 |
|---|---|---|
| 市場 | 対象市場の定義（製品カテゴリ/顧客タイプ/バリューチェーン上の範囲） | 「○○市場」の○○が業界標準の定義か自前定義かを明記していなければ不合格 |
| 期間 | 実績データの参照期間と、予測の対象期間 | 「直近」「最近」等の相対語のみはNG。西暦年で書く |
| 地域 | 国/リージョン。為替換算する場合はレートと基準日 | 換算レート未記載の外貨→円換算は不合格 |
| セグメント | B2B/B2C、企業規模、年齢層、価格帯等の切り口 | セグメント無指定なら「全体」と明記する（暗黙にしない） |

## 2. 情報源の信頼度階層（Source Tier）

### 2.1 5段階のTier定義

| Tier | 定義 | 例 |
|---|---|---|
| **Tier 1** | 一次情報・公的統計 | 政府統計（e-Stat、経産省、総務省、Census等）、決算資料・有価証券報告書、IR資料、企業の公式発表（プレスリリース原文）、特許・登記、自社データ、当事者への直接ヒアリング |
| **Tier 2** | 業界レポート・学術 | 調査会社レポート（矢野経済研究所、富士キメラ総研、Gartner、IDC、Statista等）、査読付き論文、業界団体の統計 |
| **Tier 3** | 大手報道 | 日経、NHK、Reuters、Bloomberg、WSJ、TechCrunch等の編集部があるメディアの報道記事 |
| **Tier 4** | 企業ブログ・業界メディア | 企業のオウンドメディア、note、業界系Webメディア、カンファレンス講演資料、Wikipedia |
| **Tier 5** | SNS・UGC・匿名情報 | X（Twitter）、Reddit、5ch、レビューサイトの個別投稿、匿名ブログ、出典不明のまとめ記事 |

### 2.2 Tier使用ルール（判定文）

- T-01: **Key Findings（主要な発見）の根拠は Tier 3 以上に限る。** Tier 4-5 のみを根拠とする事実を Key Findings に昇格させてはならない。
- T-02: Tier 4-5 は「**兆候・仮説の材料**」としてのみ使用する。レポートに載せる場合は「〜という声がある（Tier 5・N件確認）」の形式で、事実ではなく兆候として書く。
- T-03: 市場規模・シェア・成長率など**意思決定を左右する数値は Tier 1-2 を必須**とする。Tier 3 の報道が数値を引用している場合は、報道が引いている元レポート（Tier 1-2）まで遡って確認する（§3.3）。
- T-04: Wikipedia は Tier 4。直接引用禁止。脚注にある一次ソースへ遡って、そのソースのTierで扱う。
- T-05: 生成AIの出力・出典のない要約サイトは**Tier外（使用禁止）**。出典として記録すること自体を禁じる。
- T-06: 同一企業の自社発表（Tier 1だが利害当事者）を競合比較の唯一の根拠にしない。第三者ソース（Tier 2-3）で相対化する。
- T-07: Tierの判定に迷ったら**低い方のTierを採用**する（保守的判定の原則）。

### 2.3 Tierと確信度の対応目安

| 根拠の状態 | 記載する確信度 |
|---|---|
| Tier 1-2 の2ソース以上が一致 | 高 |
| Tier 1-2 の単一ソース、または Tier 3 の複数ソース | 中 |
| Tier 3 単一、または Tier 4-5 のみ | 低（Key Findings不可） |

この確信度は Step 5（仮説立案）の確信度表記にそのまま接続する。

## 3. WebSearch / WebFetch の実務手順

### 3.1 検索クエリ設計

1回の調査テーマにつき、**最低でも以下の4系統のクエリを設計**してから検索を始める。

| 系統 | 目的 | クエリ例 |
|---|---|---|
| 日本語クエリ | 国内情報の網羅 | `○○市場 市場規模 2026` |
| 英語クエリ | 海外・グローバル情報 | `○○ market size CAGR 2026 Japan` |
| site:指定クエリ | Tier 1-2 への直行 | `○○ site:meti.go.jp` / `○○ 決算 site:ssl4.eir-parts.net OR site:irbank.net` |
| 期間指定クエリ | 鮮度の担保 | 検索ツールの期間フィルタで「過去1年」を指定、またはクエリに年号を含める |

**クエリ設計ルール（判定文）**:

- Q-01: 重要RQ（優先度High）は日英両方で検索する。日本語のみで済ませた場合、その旨を調査ログに残す。
- Q-02: 1つのRQに対して検索クエリを3回変えても Tier 3 以上のソースが見つからない場合、そのRQは「要調査（未充足）」と記録して先へ進む（粘着しない）。
- Q-03: 検索結果の1ページ目タイトルだけで引用しない。**引用する前に必ず WebFetch で本文を読む。**

### 3.2 クロスチェック（数値の照合）

- X-01: **レポートに載せる重要数値（市場規模・シェア・成長率・価格・ユーザー数）は、独立した2ソース以上で照合する。** 同じ元レポートを引用している2記事は「1ソース」と数える。
- X-02: 2ソースの数値が乖離する場合: 乖離が±20%以内なら幅で記載（「約800〜950億円」）。±20%超なら両論併記し、定義の違い（市場の範囲・年度・為替）を調べて注記する。
- X-03: 照合できなかった数値は「単一ソース・未照合」と脚注に明記する。照合済みのふりをしてはならない。

### 3.3 一次ソースへの遡り方

報道・ブログが数値を出している場合、以下の手順で遡る。

1. 記事内の「〜社の調査によると」「〜省統計」等の言及を特定する。
2. 元レポート名・統計名で再検索し、発行元の公式ページを WebFetch する。
3. 元ソースが確認できたら、出典は**元ソース（高いTier）で記録**し、経由した記事はログに残す。
4. 元ソースが有料で本文確認できない場合: 出典は元レポート名を記載しつつ「二次引用（原典未確認）」と注記する。原典確認済みと偽装しない。

## 4. 出典管理

### 4.1 収集時の必須記録項目

情報を1件収集するたびに、以下の7項目を**その場で**記録する（後からまとめて書かない — URLと日付の記憶違いが捏造の温床になる）。

| # | 項目 | 例 |
|---|---|---|
| 1 | 媒体（発行元） | 経済産業省 |
| 2 | タイトル | 「令和7年 特定サービス産業動態統計」 |
| 3 | 発行日（不明なら「発行日不明」） | 2026-03-15 |
| 4 | URL | https://... |
| 5 | 参照日（自分がアクセスした日） | 2026-07-11 |
| 6 | 信頼度Tier | Tier 1 |
| 7 | 引用箇所（ページ/セクション/該当の一文） | p.12 表3「市場規模 842億円（2025年）」 |

**判定**: 7項目のうち1つでも欠けた収集メモは「出典タグ付き」と見なさない（Step 2 完了判定にカウントできない）。

### 4.2 レポートへの反映

- 本文中は `OUTPUT_RULES.md` 準拠の脚注形式で出典を付ける（形式の詳細は同ファイルに従う）。
- 脚注には最低限「媒体・タイトル・発行日・参照日」を含める。URLがある場合はリンクにする。
- 「事実」ラベル（§5.1）の記載には**1文ごとではなく主張ごと**に脚注を付ける（同一ソースの連続引用は段落末に1つでよい）。

## 5. 数値の扱い

### 5.1 「事実 / 推定 / 仮定」の3分類

Step 3（情報整理）のラベルに対応する。収集・記載の時点で必ずいずれかに分類する。

| 分類 | 定義（判定文） | レポート上の表記 |
|---|---|---|
| **事実** | Tier 1-3 のソースで確認でき、出典7項目が揃っている | 数値+脚注。「（出典: …）」 |
| **推定** | ソースの数値から計算・按分・外挿した値、またはフェルミ推定 | 「推定○○（算出根拠: …）」と算出式を明記 |
| **仮定** | 検証していない前提として置いた値 | 「仮定: ○○とする」と前提条件章に明記 |

- N-01: 3分類のいずれにも当てはまらない数値（＝どこから来たか説明できない数値）をレポートに書くことを禁止する。
- N-02: 「推定」の算出式を書けない場合、それは推定ではなく捏造である。書けるまで載せない。

### 5.2 フェルミ推定の作法

- 前提を**式で**示す。例: `国内SOM ≒ 対象企業数 12万社 × 想定導入率 3% × ARPU 24万円/年 = 約86億円`
- 式の各因数について「事実/推定/仮定」のラベルと出所を1行ずつ添える。
- 感度が高い因数（±で結果が2倍以上変わるもの）は、悲観/中位/楽観の3ケースで幅を示す。
- フェルミ推定の結果を Tier 1-2 の実データと同格に扱わない。「推定」表記を外してはならない。

### 5.3 古いデータの扱い

- D-01: すべての数値に**基準日（データが指す時点）**を明記する。「現在」「最新」という語で時点をぼかさない。
- D-02: 基準日から**2年以上**経過したデータを意思決定の根拠に使う場合、鮮度リスクをレポートの Risks 章に記載する（`SKILL.md` §3 エラーハンドリング準拠）。
- D-03: より新しいデータの不在を確認したうえで古いデータを使う場合、「YYYY年時点が確認できる最新」と注記する。

### 5.4 捏造の絶対禁止

- F-01: **確認できない数値・存在しないURL・実在しないレポート名を生成することを、いかなる理由でも禁止する。**
- F-02: 数値が見つからないとき正しい行動は次の3つのみ: (a) 「要調査」と明記して空欄にする勇気を持つ (b) §5.2 の作法でフェルミ推定に切り替え「推定」と明記する (c) スコープから外し前提条件に明記する。
- F-03: 「もっともらしい精緻な数値」（例: 843.7億円）を根拠なく書くことは、丸めた数値を書くより悪い。精度の見た目は信頼性ではない。
- F-04: Step 11 のチェックで出典を後付けできなかった数値は、削除または「要調査」に置換する。

## 6. 定性調査（口コミ・レビュー・SNS分析）

### 6.1 サンプリング方法

- S-01: 単一プラットフォームで完結させない。**最低2プラットフォーム**（例: App Store + Google Play、X + レビューサイト）から収集する。
- S-02: 収集件数の目安: 1テーマにつき**30件以上**（30件未満の場合は「サンプル僅少（N=xx）」と明記し、傾向の断定をしない）。
- S-03: 収集条件（期間・検索語・並び順）を記録する。「並び順: 参考になった順」と「新着順」では偏りが異なるため、可能なら両方から取る。
- S-04: 星評価があるプラットフォームでは、高評価・低評価の両側から取る（片側のみの収集を禁止）。

### 6.2 ポジ/ネガ/ニュートラル分類

- 収集した声を「ポジティブ / ネガティブ / ニュートラル（要望・質問含む）」に分類し、**件数と比率**で報告する（例: ポジ 42% / ネガ 35% / 中立 23%、N=60）。
- 1件の投稿に賛否が混在する場合は主旨で判定し、判定に迷うものは中立に落とす（保守的判定）。
- 分類結果はさらにテーマ別（価格/機能/サポート/UX等）に集計し、**上位3テーマ**をレポートに載せる。

### 6.3 代表的な声の抽出ルール — チェリーピッキングの禁止

- C-01: 引用する「代表的な声」は、§6.2 の集計で**件数上位のテーマから**選ぶ。少数意見を引用する場合は「少数意見（N=2/60）」と明記する。
- C-02: 自分の仮説に都合の良い声だけを引用することを禁止する。ネガ比率が35%なら、引用にもネガを含める。
- C-03: 引用は原文の趣旨を変えない範囲で短縮し、個人が特定できる情報（アカウント名等）は伏せる。
- C-04: SNSの声はすべて Tier 5 として扱う（§2.2 T-02: 兆候としてのみ使用）。

## 7. 競合調査の標準項目リスト

競合1社につき以下の9項目を調査する。埋まらない項目は「要調査」と記載する（空欄のまま表を出さない）。

| # | 項目 | 調べる内容 | 主な情報源（Tier） |
|---|---|---|---|
| 1 | 価格 | 料金プラン、価格帯、無料枠、値上げ履歴 | 公式サイト（1）、報道（3） |
| 2 | 機能 | 主要機能一覧、直近1年のリリース、ロードマップ公表有無 | 公式ドキュメント（1）、リリースノート（1） |
| 3 | UX | オンボーディング、主要導線、アプリ評価スコア | 実際に触る（1）、ストアレビュー（5） |
| 4 | ブランド | ポジショニングメッセージ、受賞、指名検索の傾向 | 公式（1）、報道（3） |
| 5 | 口コミ | §6 の方法でポジ/ネガ比率と上位テーマ | レビュー・SNS（5） |
| 6 | 課金導線 | フリーミアム/トライアル/営業経由、アップセル設計 | 公式（1）、実際に触る（1） |
| 7 | SNS | 公式アカウントのフォロワー数・投稿頻度・エンゲージの傾向 | SNS公式（1）※数値は参照日を明記 |
| 8 | 採用動向 | 募集職種と人数感（投資領域のシグナル） | 採用ページ（1）、求人媒体（4） |
| 9 | 資金調達 | 調達額・ラウンド・投資家、上場企業なら業績 | IR・登記（1）、報道（3） |

- 比較する競合は**3〜5社**を標準とする（2社以下は比較にならず、6社以上は表が読めなくなる。超える場合は「主要比較3〜5社+ロングリスト」の2段構成にする）。

## 8. 調査の終了判定

以下の**いずれか**を満たしたら調査を終了し、Step 3 へ進む。

| 判定 | 基準（判定文） |
|---|---|
| **飽和** | 新しい情報源を3つ連続で確認しても、既存メモにない新事実が1件も出ない |
| **充足** | Key Findings 候補が10件に到達し、全件が出典7項目付き・優先度HighのRQがすべて充足している |

追加ルール:

- E-01: 上記を満たさないまま調査を打ち切る場合（時間制約等）、未充足のRQを列挙し、レポートの「前提条件」と「Risks」に「調査未了事項」として明記する。
- E-02: 終了判定を満たした後の追加調査は、Step 4 以降で新たな問いが生じた場合のみ行う（際限ない調査の防止）。
- E-03: Key Findings 候補10件のうち、Tier 3 以上を根拠とするものが7件未満の場合は充足と見なさない（T-01 との整合）。

## 9. 調査ログの残し方

レポート本文とは**別ファイル**で調査メモを構造化して残す。目的は (a) Step 11 での出典検証 (b) 追加調査時の重複防止 (c) スキル改善ループへの還元。

### 9.1 保存先と命名

- 保存先: レポートと同じディレクトリの `_research/` 配下、またはユーザー指定の場所。
- 命名: `YYYY-MM-DD_{report-slug}_research-log.md`
- 調査ログはレポートの成果物ではないため、共有時に含めるかはユーザーに確認する。
