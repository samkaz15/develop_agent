## 1. タイポグラフィ

### 1.1 見出し階層ルール

| レベル | 用途 | 個数制限 | 判定 |
|---|---|---|---|
| H1 (`#`) | 文書タイトル | 文書全体で**1つのみ** | `#` で始まる行が2行以上あればNG |
| H2 (`##`) | 章（TEMPLATE.mdの12章） | 3〜12個 | 章以外にH2を使ったらNG |
| H3 (`###`) | 節（章内の区分） | 1章あたり0〜5個 | 6個以上なら章を分割する |
| H4以下 (`####`〜) | 原則禁止 | 例外は下記1点のみ | — |

- H4の唯一の例外: `8. Strategic Recommendations` 内の施策見出し（`#### 施策N: 名称`）。それ以外でH4が必要になったら、構造が深すぎるサイン。表または箇条書きに落とす
- H5・H6は例外なく禁止
- 見出しレベルの飛ばし禁止: H2の直下にH4を置かない（H2→H3→H4の順を守る）

### 1.2 見出しの文字数上限

| レベル | 上限 | 超過時の対応 |
|---|---|---|
| H1（タイトル） | 全角35文字 | サブタイトルに分割する |
| H2（章） | 全角25文字 | 章名を短縮し、説明は本文第1文へ |
| H3（節） | 全角20文字 | 節名を体言止めに短縮 |

- 見出しの末尾に句点（。）・コロン（:）を付けない
- 見出しに太字・イタリック・リンクを含めない（GitHubのアンカー生成が不安定になるため）

### 1.3 章番号規則

- H2は「番号 + 半角ピリオド + 半角スペース + 章名」: `## 8. Strategic Recommendations`
- H3は「章番号.節番号」: `### 8.1 戦略オプション比較`（節番号の後ろにピリオドを付けない）
- 章を省略した場合も番号は1から連番で詰める（欠番禁止。TEMPLATE.md §マトリクスの運用ルール4と整合）

## 2. タイトル/サブタイトル設計

### 2.1 良いタイトルの型

**原則: 結論を含むタイトル > テーマだけのタイトル**。読者は一覧画面（GitHubのファイルリストや通知）でタイトルだけを見て開くか判断する。

| 型 | 判定 | 例 |
|---|---|---|
| ✅ 結論型 | 主張（動詞・評価・数値）を含む | 「国内オンライン学習市場は2028年に3,200億円 — 法人向けセグメントへの参入を推奨」 |
| ✅ 問い+答え型 | 意思決定事項への答えを含む | 「A案採用の提言: 競合3社比較で唯一未充足の中価格帯を獲る」 |
| ❌ テーマ型 | 対象しか書かれていない | 「オンライン学習市場調査レポート」 |
| ❌ 手法型 | 何をしたかしか書かれていない | 「競合5社の4P分析」 |

判定文: **タイトルだけ読んで「で、結論は？」と聞き返されるならNG**。

### 2.2 タイトルの構造ルール

- H1タイトル: 全角35文字以内。結論を含める。機密区分・日付は入れない（メタ情報表に書く）
- サブタイトル: H1直後に `**太字**` 1行で置く。「対象 + 期間」を書く（例: 「国内EdTech市場 2023–2028年 参入判断資料」）。全角30文字以内
- タイトルとExecutive Summaryの結論は矛盾してはならない（品質チェック時に突合する）

## 3. 余白

GitHub Markdownにおける余白は空行で制御する。

### 3.1 空行ルール

| 位置 | 空行数 | 備考 |
|---|---|---|
| H2（章）の前 | 2行（うち1行は `---` 区切り線でもよい） | 章の切れ目を視覚化する |
| H2の後 | 1行 | |
| H3の前後 | 1行 | |
| 表の前後 | 1行 | 空行がないとGFMで表がレンダリングされない |
| コードブロック・Mermaidの前後 | 1行 | 同上 |
| 箇条書きブロックの前後 | 1行 | |
| 段落間 | 1行 | 2行以上連続させない（章の前を除く） |

### 3.2 セクションの長さ上限（スクロール疲れ防止）

- 1つのH2章は**最大100行**（表・図込み）。超える場合はH3で分割するか、補足を `<details>` に折りたたむ（使用条件は `OUTPUT_RULES.md` §折りたたみ）
- 見出しなしで本文が連続してよいのは**最大15行**。超えたらH3または表・リストに構造化する
- 図表なしでテキストだけが続いてよいのは**最大20行**。超えたら図表化候補を探す（品質基準3: 主要な主張の8割以上に図表・リストの裏付け）

## 4. 表

表は本スキルの主力表現である。グラフが使えないGitHub Markdownでは、数値の主張は原則すべて表で示す。

### 4.1 サイズ上限

| 項目 | 推奨 | 上限 | 超過時の対応 |
|---|---|---|---|
| 列数 | 5列以内 | 7列 | 列を分割して2表にする、または行列を転置する |
| 行数 | 10行以内 | 10行 | 「上位N + その他」に集約するか、全量を `<details>` に格納し本文には要約表を置く |
| セル内文字数 | 全角20文字以内 | 全角40文字 | セル内改行はせず、行を分けるか本文に出す |

### 4.2 数値の書式

- 数値列はヘッダ区切り行で右揃え指定する: `|---:|`（GitHubは右揃えを反映する）
- 桁区切りカンマ必須: `12,500`（4桁以上の整数すべて）
- 1つの表の中で単位を混在させない。単位は列ヘッダに書く（例: `市場規模（億円）`）。セル内に単位を繰り返さない
- 比較列には基準を示す: 前年比は `+12%` / `-3%` のように符号を付ける
- 欠損値は空欄にせず `—`（全角ダッシュ）、未計測は `要計測`、推定は値の後に `（推定）` を付ける
- 詳細な数値表記統一は `OUTPUT_RULES.md` §数値表記統一に従う

### 4.3 ヘッダ行の書き方

- ヘッダは名詞または「名詞（単位）」。文にしない
- 主語となる列（項目名・社名等）を必ず最左列に置く
- 判定・結論を示す列（推奨/優先度等）は最右列に置く（左から右へ「事実→判定」の流れを作る）
- 太字はヘッダ行に使わない（GFMが自動で強調するため冗長）。ただし表内の「判定行」（例: TEMPLATE.md §8.1の `**判定**` 行）は例外として太字可

## 5. 図解（Mermaid・テキスト図）

### 5.1 Mermaidを使う場面/使わない場面

| 表現したいもの | 手段 | 図種 |
|---|---|---|
| フロー・因果・KPIツリー | ✅ Mermaid | `flowchart` |
| スケジュール・ロードマップ | ✅ Mermaid | `gantt`（タスク12個以内） |
| 時系列の経緯 | ✅ Mermaid | `timeline` |
| 2軸ポジショニング（象限） | ✅ Mermaid | `quadrantChart` |
| 構成比（項目5個以内） | ✅ Mermaid | `pie` |
| 数値の正確な比較 | ❌ 表を使う | 目視で値が読めない図はNG |
| 項目2個だけの関係 | ❌ 本文1文で書く | 図にする価値なし |
| 6項目超の構成比・複雑な組織図 | ❌ 表を使う | 可読性が崩壊する |

使用可能な図種は上記5種のみ（`OUTPUT_RULES.md` §Mermaid規定と整合）。

### 5.2 Mermaidの複雑度上限

| 項目 | 上限 | 超過時 |
|---|---|---|
| ノード数（flowchart） | 12個 | 表に落とす、または図を分割 |
| 階層の深さ（flowchart/ツリー） | 3階層 | 4階層目以降は表に落とす |
| ganttのタスク数 | 12個 | フェーズ表（TEMPLATE.md §10.1）に落とす |
| quadrantChartの点数 | 8個 | 上位8件に絞り残りは表 |
| pieの項目数 | 5個 | 「上位4 + その他」に集約 |
| エッジのラベル文字数 | 全角8文字 | ラベルを外し凡例を本文に書く |

判定文: **図を10秒眺めて構造が頭に入らなければ複雑すぎる**。迷ったら表に落とす（フォールバック規則は `OUTPUT_RULES.md` §Mermaid規定）。

### 5.3 テキスト図の型

Mermaidが不適な単純構造は、以下の2型のテキスト図のみ許可する（コードブロック内に置く）。

型1: 段階・ファネル（3〜5段）

```text
認知 1,000人 → 登録 300人 (30%) → 有料化 45人 (15%)
```

型2: 対比・Before/After

```text
[現状]   手動対応: 平均48時間
[施策後] 自動対応: 平均 2時間（▲96%）
```

アスキーアートによる複雑な図（罫線で箱を描く等）は禁止。それが必要ならMermaidか表を使う。

## 6. グラフ選定

GitHub Markdownはグラフ画像を自己完結で埋め込めない（外部画像リンクは `OUTPUT_RULES.md` §禁止事項）。データの型ごとに以下の代替表現を使う。

### 6.1 データの型 → 図の型 対応表

| データの型 | 理想のグラフ | GitHub Markdownでの表現 |
|---|---|---|
| 比較（項目間の大小） | 横棒グラフ | 表（降順ソート）+ バー列（下記6.2） |
| 推移（時系列） | 折れ線グラフ | 年次表 + 前年比列。3期以上あればスパークライン的表現可 |
| 構成比（全体の内訳） | 円/積み上げ棒 | 構成比表（%列 + 合計100%行）または Mermaid pie（5項目以内） |
| 相関（2変数の関係） | 散布図 | Mermaid quadrantChart（8点以内）または2軸評価表 |
| フロー・因果 | ダイアグラム | Mermaid flowchart |
| スケジュール | ガントチャート | Mermaid gantt または フェーズ表 |

### 6.2 スパークライン的表現（表内バー）

数値の大小を直感的に見せたい場合、表に「バー」列を追加してよい。

| セグメント | シェア | バー |
|---|---:|---|
| 法人向け | 45% | `█████████` |
| 個人向け | 30% | `██████` |
| 教育機関 | 25% | `█████` |

規則:

- `█`（U+2588）のみ使用。1文字 = 5%換算（最大値基準の相対でもよいが、換算規則を表の直下に1行で注記する）
- バーは10文字以内。数値列を省略してバーだけにするのは禁止（バーは補助、数値が主）
- 推移を1セルで示す場合は `▁▃▅▇`（U+2581/2583/2585/2587）の4段階のみ使用可

### 6.3 選定の判定手順

1. 読者に伝えたい結論を1文で書く（例: 「法人向けが最大セグメントである」）
2. 結論の型を上表で引き、表現を決める
3. その図表の直前または直後の本文第1文に、結論の1文をそのまま書く（**図表に語らせきりにしない**）

## 7. 色と絵文字

GitHub Markdownでは文字色を制御できない。強調手段の優先順位と絵文字規則を定める。

### 7.1 強調手段の優先順位

強い強調が必要なとき、上から順に検討する。

| 優先 | 手段 | 用途 |
|---|---|---|
| 1 | **太字** | 文中のキーワード・結論語句 |
| 2 | 引用ブロック（`>`） | 重要な注意・要約（§9の型に限る） |
| 3 | 絵文字バッジ 🔴🟡🟢 | 表・リスト内のステータス/優先度表示 |
| 4 | `コードスパン` | 数式・ファイル名・リテラル値のみ |

HTMLタグ（`<font>`, `<span style>`等）による着色は禁止（`OUTPUT_RULES.md` §禁止事項）。

### 7.2 絵文字の使用ルール

- **用途はステータス表示のみ**。装飾（見出しの飾り、感情表現、区切り）としての使用は禁止
- 使用可能な系統は以下の3系統。**1つの表につき1系統のみ**（混在禁止）

| 系統 | 絵文字 | 意味 |
|---|---|---|
| 信号 | 🔴 🟡 🟢 | 優先度 高/中/低、リスク 高/中/低、状態 危険/注意/良好 |
| 判定 | ✅ ❌ | 採用/不採用、合格/不合格、対応/未対応 |
| 矢印 | ⬆️ ⬇️ ➡️ | 指標の増加/減少/横ばい |

- 同一表内で同じ絵文字は同じ意味で使う。意味は表の直下または列ヘッダで自明にする（例: 列名を「優先度」とする）
- 1レポート内で同じ系統は同じ意味に固定する（4章で🔴=リスク高、8章で🔴=優先度高、のような揺れは表ごとに列名で意味が特定できれば許容。特定できなければNG）
- 本文（表・リスト外）での絵文字使用は禁止

## 8. 強調方法

| 手段 | ルール | 判定 |
|---|---|---|
| 太字 | **1段落につき2箇所まで**。1箇所あたり全角15文字以内 | 3箇所以上ある段落はNG。全部太字=強調ゼロと同じ |
| イタリック | **非推奨（原則使用しない）** | 日本語は斜体の視認性が低い。英語の書名・学術用語のみ例外可 |
| 引用ブロック | 「重要な注意・要約」専用（§9） | 会話の引用・段落インデント代わりの使用はNG |
| 取り消し線 | 使用禁止 | 修正履歴はレポートに残さない |
| ALL CAPS/全角強調 | 使用禁止 | 「！！」「★」等の記号強調も禁止 |

- 太字にするのは「結論語句・数値・固有名詞のキーワード」のみ。文全体の太字は1章につき1文まで（結論文に限る）
- 見出し・表ヘッダは構造上すでに強調されているため、追加の太字を重ねない（§4.3の例外を除く）

## 9. 注意書き・免責

### 9.1 注意書きの型

注意・免責・重要な留保は、以下の型の引用ブロックに統一する。

```markdown
> **注:** {{注意内容を1〜2文で}}
```

- ラベルは `**注:**`（一般的注意）、`**免責:**`（免責事項）、`**要約:**`（章要約）の3種のみ
- 1つのH2章に引用ブロックは**2個まで**。3個以上必要なら本文の構成を見直す
- 引用ブロック内は3行以内

### 9.2 推定値の表記法

- 本文・表中の推定値は、値の直後に `（推定）` を付ける: `約1,200億円（推定）`
- 推定の根拠は脚注または直後の注記に書く: `> **注:** 上記は◯◯の公開値に△△の比率を乗じた推定値である。`
- 「約」「およそ」だけで推定を示すのは禁止（推定と概数は別概念）。詳細は `WRITING_RULES.md` と `OUTPUT_RULES.md` §参考文献に従う

## 10. ページ構成

### 10.1 結論ファースト（品質基準1）

- 文書全体: 結論はExecutive Summary（第1章）にある。「最後まで読むと結論がわかる」構成は禁止
- 各章: **第1文が結論**。書き出しの型は `TEMPLATE.md` の各章定義に従う
- 各表・図: 直前または直後の本文に、その図表から言える結論を1文で書く（§6.3）

判定文: 各H2の直後の1文だけを12個拾い読みして、レポートの主張が再構成できればPass。

### 10.2 1章1メッセージ

- 1つのH2章で主張してよい結論は**1つ**。2つ以上の独立した結論が必要なら章を分けるか、Key Findingsに番号付きで分離する
- 章の結論と無関係な情報（面白いが判断に効かない事実）は削除するか `<details>` へ（品質基準5: 意思決定に必要な情報だけ）

### 10.3 目次

- **H2が4章以上あるレポートは目次必須**（3章以下は省略可）
- 目次はメタ情報表の直後、第1章の前に置く
- 番号付きリスト + アンカーリンク形式（アンカー規則は `OUTPUT_RULES.md` §Markdown/GitHub最適化）
- 目次に載せるのはH2のみ（H3は載せない）

## 11. 文章量

| 項目 | 上限 | 判定 |
|---|---|---|
| 1段落 | 3文以内 | 4文目が必要なら段落を割る |
| 1文 | 全角60文字以内（目安） | 80文字を超える文は必ず分割する |
| 箇条書き1項目 | 1行（全角40文字以内） | 2行に折り返すなら「太字ラベル — 説明」形式に変えるか本文へ |
| 箇条書きのネスト | 2階層まで | 3階層目が必要なら表にする |
| 箇条書きの項目数 | 1ブロック7個まで | 8個以上は表にするか分類してH3を切る |

- 推奨分量は `TEMPLATE.md` の各章定義に従う。全章合計の本文目安は種別により400〜1,000行
- 「書けるだけ書く」は禁止。分量の上限超過は品質基準5（意思決定に必要な情報だけ）への違反として扱う

## 12. トーン&マナー

### 12.1 基本トーン（経営層向け）

- **敬体（です・ます）または常体（である）を文書内で統一**する。既定は常体（である調）。読者が社外の場合のみ敬体
- **断定的に書く**: 事実と分析に基づく主張は「〜と考えられる可能性もある」ではなく「〜である」「〜を推奨する」と書く。不確実性は確信度・推定表記で明示的に扱う（ぼかし表現で扱わない）
- **数値中心**: 形容詞による評価（「大きい」「急成長」）には必ず数値を併記する。判定文: 数値の裏付けのない評価形容詞が1文に1つでもあればNG

### 12.2 禁止トーン

| 禁止 | 例（NG） | 代替（OK） |
|---|---|---|
| 煽り | 「今すぐ参入しないと手遅れに！」 | 「参入判断の期限は2026-12末である（理由: …）」 |
| 過度な感嘆符 | 「売上2倍を達成！！」 | 「売上は前年比+100%となった」（感嘆符は全文書で0個） |
| 自画自賛 | 「弊社の圧倒的な技術力により」 | 「自社の◯◯（特許N件）により」 |
| 過剰な謙遜・ぼかし | 「あくまで私見ですが」「〜かもしれません」 | 「確信度: 中。根拠は…」 |
| 修辞的疑問 | 「この機会を逃す手はあるだろうか？」 | 疑問文はPurposeの意思決定事項のみ許可 |

### 12.3 用語

- 専門用語・略語は初出時に補足を付ける（品質基準5）: `CAC（顧客獲得コスト）`。2回目以降は略語のみでよい
- 初出補足が必要な用語が10個を超える場合は、末尾に用語集（表）を置き、初出補足は5個以内に抑える

## 13. Learned Principles

### 13.1 追記ルール

1. 追記できるのは**デザイン思想のみ**（レイアウト・情報設計・章構成・図表の使い方・余白・文字量・タイトル設計・色・グラフ選定・ストーリー構成・要約方法）。参考資料の文章・データ・図表内容の転記は禁止
2. 追記する原則は本ファイルの他の規則と同じく**判定可能な形式**（数値・判定文）に変換してから書く。変換できない感想（「余白が美しい」等）は追記しない
3. 1原則 = 表の1行。既存規則と矛盾する場合は既存規則を改訂し、本表には「§Xを改訂」と記録する
4. 追記時は本ファイル冒頭の Version をパッチ更新（例: 1.0.0 → 1.0.1）し、Last Updated を更新する
