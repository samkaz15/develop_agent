# Marketing Report Skill

経営層・投資家にそのまま提出できる品質のレポートを、**誰が実行しても同一品質**で生成するためのスキル。
本スキルは単発のレポート生成ツールではなく、**再現性のあるレポート生産システム**である。

## 0. 実行前に必ず行うこと（Progressive Disclosure）

本スキルの知識は複数ファイルに分割されている。**一度に全部読まず、下表の「読むタイミング」に従って必要時にReadする**こと。

| ファイル | 役割 | 読むタイミング |
|---|---|---|
| `TEMPLATE.md` | レポートの標準12章構成と各章の書式 | Step 10（レポート生成）の直前に必読 |
| `STYLE_GUIDE.md` | タイポグラフィ・余白・表・図・色・トーンの規定 | Step 10 の直前に必読 |
| `WRITING_RULES.md` | 文章規則（結論ファースト・数値表記・禁止表現） | Step 10 の直前に必読 |
| `RESEARCH_METHOD.md` | 調査の手順・情報源の信頼度評価・出典管理 | Step 2（調査）の直前に必読 |
| `ANALYSIS_FRAMEWORK.md` | 14フレームワークの定義と選択マトリクス | Step 3〜6 の直前に必読 |
| `OUTPUT_RULES.md` | Markdown/GitHub最適化・Mermaid・脚注・メタ情報の出力規定 | Step 10 の直前に必読 |
| `CHECKLIST.md` | 品質チェック9領域の合格判定基準 | Step 11（品質チェック）で必読 |
| `PROMPTS.md` | 分析タイプ別の再利用プロンプト16種 | 該当する分析を行うときのみ |
| `EXAMPLES.md` | 良い例・悪い例・改善例 | 品質判断に迷ったときのみ |
| `assets/README.md` | 参考資料の追加・解析ルール | Step 0-B（参考資料の解析）で必読 |
| `assets/reference/` | ユーザーが追加した参考資料（PDF/PPT/画像等） | Step 0-B で存在すれば解析 |

## 0-A. このスキルが適用されるレポート種別

市場調査 / 競合分析 / UX分析 / プロダクト分析 / SNS・SEO・口コミ分析 / 価格・LTV・CAC等の指標分析 / 投資家向け資料 / 経営会議資料 / 事業計画サマリー。
種別が上記にない場合も、`TEMPLATE.md` の12章構成をベースに章を取捨選択して適用する（削除した章と理由をレポート冒頭の「前提条件」に明記する）。

## 0-B. 参考資料の解析（存在する場合のみ）

`assets/reference/` にファイルがある場合、レポート作成前に解析する。

**学習してよいもの（デザイン思想のみ）**: レイアウト、情報設計、章構成、見出し構成、図表の使い方、余白、文字量、タイトル設計、色の考え方、グラフ選定、アイコン利用、ストーリー構成、経営者/投資家向け表現、要約方法、結論のまとめ方。

**絶対に行わないこと**: 参考資料の文章・データ・画像・図表の内容をコピー、転載、翻案すること。参考資料は「どう見せるか」を学ぶためのものであり、「何を書くか」の情報源ではない。詳細は `assets/README.md` を必読。

## 1. 実行フロー（11 Steps — 省略禁止）

以下の11ステップを必ずこの順で実行する。各ステップには**完了判定**があり、満たさない限り次へ進まない。

### Step 1: 目的確認

- 確認する項目: ①レポートの種別 ②読者（経営層/投資家/実務者/社内共有）③意思決定してほしいこと ④スコープ（対象市場・期間・地域）⑤納期と分量の希望 ⑥既存の前提資料の有無
- ユーザーの依頼文から埋まらない項目は**推測せず質問する**（自律実行時は合理的なデフォルトを置き、レポートの「前提条件」に明記する）
- **完了判定**: 上記6項目がすべて「確定」または「前提として明記」の状態になっている

### Step 2: 調査

- `RESEARCH_METHOD.md` をReadしてから実行する
- 一次情報 > 公的統計 > 業界レポート > 報道 > ブログ/SNS の信頼度順で収集し、全事実に出典・調査日時を付ける
- 数値が確認できないものは「推定」と明示する（架空の精緻な数値・偽の出典URLの生成は禁止）
- **完了判定**: Key Findingsの候補となる事実が10件以上、各々に出典タグが付いている

### Step 3: 情報整理

- 収集情報を「事実 / 解釈 / 推定」の3ラベルに分類する
- MECEにグルーピングし、レポートの章（`TEMPLATE.md` の4〜6章）に仮配置する
- **完了判定**: すべての収集情報に3ラベルのいずれかが付き、重複が除去されている

### Step 4: インサイト抽出

- `ANALYSIS_FRAMEWORK.md` の選択マトリクスで、目的に合うフレームワークを1〜3個選ぶ（多用しない。選定理由を書けないフレームワークは使わない）
- 「事実の羅列」と「インサイト」を区別する: インサイトとは **So What（だから何か）が言えて、意思決定を変えうる発見** のこと
- **完了判定**: 「事実→解釈→So What」の3段が揃ったインサイトが3〜7個ある

### Step 5: 仮説立案

- 各インサイトから「もし〜すれば、〜になるはずだ」形式の検証可能な仮説を立てる
- 仮説には確信度（高/中/低）と、検証方法を必ず添える
- **完了判定**: 主要仮説3個以上に確信度と検証方法が付いている

### Step 6: 戦略立案

- 仮説から戦略オプションを2〜3案作り、評価軸（インパクト/実現性/リスク/投資額）で比較し、推奨案を1つに絞る
- 「推奨しない案」も比較表に残す（経営層は選択肢の比較を求める）
- **完了判定**: 推奨案が1つに確定し、比較表と推奨理由が存在する

### Step 7: 改善案作成

- 戦略を実行可能な施策に分解する（各施策: 内容 / 期待効果 / 担当想定 / 工数規模 S・M・L）
- ICEまたはRICEで優先順位を付ける（採点基準は `ANALYSIS_FRAMEWORK.md` §優先順位付け）
- **完了判定**: 施策が5個以上、全件にスコアと優先順位が付いている

### Step 8: KPI整理

- 目標をKPIツリーに構造化する（最上位目標 → 先行指標 → 施策別指標）
- 各KPI: 定義式 / 現状値（不明なら「要計測」）/ 目標値 / 計測方法 / 確認頻度
- **完了判定**: KPIツリーが1枚の図（Mermaidまたは表）で表現されている

### Step 9: ロードマップ作成

- 時間軸（フェーズまたは四半期）で施策を配置し、マイルストーンと判定条件（Go/No-Go基準）を置く
- 依存関係があるものは明示する
- **完了判定**: 全施策がロードマップ上に配置され、フェーズ移行の判定条件が数値で書かれている

### Step 10: レポート生成

- **生成直前に `TEMPLATE.md` / `STYLE_GUIDE.md` / `WRITING_RULES.md` / `OUTPUT_RULES.md` の4本を必ずRead**する
- `TEMPLATE.md` の12章構成に従い、Step 2〜9の成果を配置する
- Executive Summaryは**最後に書く**（本文完成後、全体を1ページに凝縮する）
- 出力先: ユーザー指定がなければ `docs/Reports/YYYY-MM-DD_{report-slug}.md`
- **完了判定**: 12章（または取捨選択を明記した構成）がすべて埋まり、TODO・プレースホルダが残っていない

### Step 11: 品質チェック

- `CHECKLIST.md` をReadし、9領域すべてをセルフレビューする
- 不合格項目は修正してから再チェックする（チェック結果をユーザーへの完了報告に含める）
- **完了判定**: `CHECKLIST.md` の必須項目がすべて合格（Pass）

## 2. 品質基準（毎回満たすこと）

| # | 基準 | 判定方法 |
|---|---|---|
| 1 | 結論ファースト | 各章の第1文が結論になっている |
| 2 | 要約がある | Executive Summaryが1ページ以内で「結論・根拠・推奨・数字」を含む |
| 3 | 図表中心 | 主要な主張の8割以上が表・図・リストで裏付けられている |
| 4 | 論理的 | 事実→解釈→推奨の連鎖が追跡でき、飛躍がない |
| 5 | 経営層向け | 専門用語に補足があり、意思決定に必要な情報だけがある |
| 6 | 投資家向け対応 | 市場規模・成長性・リスクが数値で語られている |
| 7 | 実務で使える | Next Actionsが「誰が・何を・いつまでに」の粒度である |
| 8 | 出典明示 | 外部データすべてに出典・調査日時があり、推定は推定と明記 |
| 9 | そのまま共有可能 | 誤字・表記ゆれ・リンク切れ・未完成箇所ゼロ |

## 3. エラーハンドリング

| 状況 | 対応 |
|---|---|
| 情報が不足していて調査で埋まらない | 「推定」と明記して仮置きし、レポートの「前提条件」と「Risks」に記載。捏造しない |
| ユーザーの目的が曖昧 | Step 1で質問する。自律実行時はデフォルト前提を置き明記する |
| 参考資料が読み取れない形式 | 解析をスキップし、その旨を報告。変換方法を `assets/README.md` に従い案内 |
| 分量指定が短くテンプレ12章が入らない | 章を統合し、省略した章名と理由を「前提条件」に明記 |
| 競合・市場の数値情報が古い | データの基準日を明記し、鮮度リスクをRisksに記載 |

## 4. スキル自体の改善ループ（AIが継続的に改善できる構造）

- レポート完成ごとに、`CHECKLIST.md` で不合格→修正した項目を振り返り、**同じ指摘が2回発生したらルール側（該当する .md）を改訂する**（失敗を基準に還元する — リポジトリ思想準拠）
- 優れたレポートが完成したら、構成・図表の型を `EXAMPLES.md` に追記する（内容は匿名化・抽象化する）
- 参考資料が追加されたら `assets/README.md` の解析ルールに従い、学んだデザイン原則を `STYLE_GUIDE.md` に還元する（資料の内容はコピーしない）
- 本スキルの変更はセマンティックバージョニングで管理し、本ファイル冒頭のVersionを更新する
