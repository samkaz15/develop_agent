## 品質哲学

### 品質の定義

> **品質とは、ユーザーが目的を達成でき、チームが自信を持って変更でき、事業が信頼を失わない状態である。**

この定義から、品質は3つの視点で測る:

| 視点 | 問い | 主な基準 |
|---|---|---|
| **ユーザー品質** | ユーザーは迷わず・速く・安心して目的を達成できるか | UX・UI・Performance・Accessibility |
| **エンジニアリング品質** | 半年後の自分・他人が安全に変更できるか | コード・テスト・ドキュメント |
| **事業品質** | ユーザーと社会の信頼を裏切らないか | Security・Privacy・倫理・法令 |

### 品質5原則

1. **判定可能でなければ基準ではない** — 「高品質」「使いやすい」は基準ではない。誰が判定しても同じ結果になる文・数値で書く。
2. **品質は後付けできない** — アクセシビリティ・セキュリティ・性能は最後に足すものではなく、最初から組み込む（Shift Left）。
3. **証拠なき合格はない** — PASS判定には計測結果・テスト結果・チェックリストの証拠を必ず添える。「見た感じ大丈夫」は判定ではない。
4. **基準を下回るなら出さない、ただし基準は現実的に** — 守れない理想値は形骸化する。基準の変更は隠れて妥協せず、PRで正式に改訂する。
5. **最低ラインは自問で終える** — 「自分がこのプロダクトのユーザー・後任者だったら満足か？」に Yes と言えないものは提出しない。

## 共通品質基準（全成果物）

ドキュメント・デザイン・コードを問わず、すべての成果物が満たすべき5基準。
（[`templates/Agent_Base_Template.md`](../templates/Agent_Base_Template.md) セクション10と同一体系）

| # | 基準 | 判定文 | 参考水準 |
|---|---|---|---|
| 1 | **明瞭さ** | 前提知識のない読み手が1回読んで理解できる。専門用語には説明がある | Stripe Docs, Notion |
| 2 | **簡潔さ** | 削れる要素をすべて削った状態。装飾より情報密度を優先している | Apple HIG, Linear |
| 3 | **一貫性** | 用語・構成・粒度・命名がリポジトリ全体で統一されている | Google Material Design |
| 4 | **検証可能性** | 主張にデータ・出典・根拠が紐づき、推測には「仮説」と明記されている | Nielsen Norman Group |
| 5 | **誠実さ** | 制約・リスク・未解決事項・トレードオフを隠さず明記している | Anthropic, OpenAI |

**形式基準（全Markdown成果物共通）**:
- [ ] 1ファイル1トピック（肥大化したら分割）
- [ ] ファイル名は英語ケバブケース
- [ ] リンク切れ・誤字脱字・フォーマット崩れがない
- [ ] Version / 更新日 / 変更履歴がある（長期運用ドキュメントの場合）

## 領域別品質基準

### 1. Documentation（ドキュメント品質）

| 基準 | 判定文 |
|---|---|
| 読者特定 | 第一読者（次工程のAgent・人間）が明記され、その読者が追加質問なしで作業を開始できる |
| 構造 | 結論が先頭にあり、詳細は後段。目次または見出しで全体像が3秒で掴める |
| 決定の記録 | 「何を決めたか」に「なぜか・何を却下したか」が添えられている |
| 鮮度 | 実装・デザインと矛盾していない（変更時に同期更新されている） |

### 2. UX品質

| 基準 | 判定文 | 検証方法 |
|---|---|---|
| タスク完遂 | ペルソナが主要タスクを迷わず完遂できる | ユーザビリティテスト（5人） |
| 異常系網羅 | エラー・空・ローディング・オフラインが全主要フローで設計されている | 状態マトリクスチェック |
| 認知負荷 | 1画面の主要アクションは1つ。選択肢・入力項目が最小化されている | ヒューリスティック評価 |
| 回復可能性 | 誤操作から1アクションで回復できる（Undo・確認・キャンセル） | フロー検査 |
| 倫理 | ダークパターン（解約妨害・強制継続・誤認誘導）を含まない | 人間レビュー |
| 根拠 | 設計判断がリサーチの発見に紐づいている | トレーサビリティ確認 |

### 3. UI品質

| 基準 | 判定文 | 検証方法 |
|---|---|---|
| システム化 | 全画面がデザイントークン・コンポーネント参照で構築されている（一点物・ハードコードなし） | Figma検査 |
| 状態網羅 | 全コンポーネントに default / hover / focus / disabled / error / empty / loading がある | 状態マトリクス |
| ガイドライン準拠 | 対象プラットフォームの規約（HIG / Material）に準拠。逸脱には理由記録がある | 準拠チェックリスト |
| レスポンシブ | 全ブレークポイントで崩れなく表示される | 実機確認 |
| 実装可能性 | Figmaファイルが実装者の仕様として機能する（命名・Auto Layout・Code Connect） | Frontend Agentレビュー |

### 4. Accessibility品質

| 基準 | 判定文 | 検証方法 |
|---|---|---|
| WCAG準拠 | WCAG 2.2 AA を満たす（機械検証＋手動検証） | axe + 手動 |
| コントラスト | テキスト 4.5:1 以上（大文字 3:1 以上） | 機械検証 |
| ターゲット | タッチターゲット 44pt（iOS）/ 48dp（Android/Web）以上 | 機械検証 |
| キーボード | キーボードのみで主要フローを完遂できる。フォーカス順序が論理的 | 手動検証 |
| 支援技術 | スクリーンリーダーで主要フローの内容・状態が伝わる | 手動検証 |
| 配慮 | Reduce Motion・Dynamic Type（対象プラットフォームの場合）に対応 | 実機確認 |

### 5. Code品質（Frontend / Backend共通）

| 基準 | 判定文 | 検証方法 |
|---|---|---|
| 規約準拠 | プロジェクトの開発規約・Lint・フォーマッタに完全準拠 | CI |
| 型安全 | 型エラーゼロ（TypeScript strict等）。`any` の濫用がない | CI |
| テスト | 変更にはテストが伴う。CIがグリーン | CI |
| エラー処理 | すべての失敗しうる処理に妥当なエラーハンドリングがある。握りつぶしがない | コードレビュー |
| 可読性 | 命名が意図を表し、コメントは「なぜ」を書いている（「何を」の重複コメントなし） | コードレビュー |
| サイズ | PRは1目的・レビュー可能なサイズに分割されている | レビュー運用 |

**Backend追加基準**:
- [ ] 全エンドポイントに認証・認可・入力バリデーションがある
- [ ] エラーレスポンス形式が統一されている
- [ ] 構造化ログが出力され、秘密情報がログに含まれない
- [ ] マイグレーションで環境を再現できる

**Frontend追加基準**:
- [ ] デザイントークン参照で実装（ハードコードなし）
- [ ] 全状態（loading / error / empty）が実装されている
- [ ] コンソールエラー・警告がゼロ
- [ ] セマンティックHTML・ARIA が適切

### 6. AI品質

| 基準 | 判定文 | 検証方法 |
|---|---|---|
| 評価駆動 | 評価データセットと合格基準（数値）がプロンプトより先に定義されている | 評価計画レビュー |
| 基準達成 | 自動評価で合格基準を達成している | 評価パイプライン |
| 回帰検知 | プロンプト・モデル変更時に評価が再実行される | CI / 評価ログ |
| フォールバック | 全AI呼び出しに失敗時のUX（タイムアウト・エラー・代替手段）がある | 障害注入テスト |
| 安全性 | プロンプトインジェクション・有害出力・個人情報漏洩への対策が実装・検証されている | セキュリティ検査 |
| コスト | トークンコストに上限とアラートがある | コスト監視 |
| 透明性 | AIが生成した内容であることがユーザーに適切に伝わる | UXレビュー |

### 7. Test品質

| 基準 | 判定文 |
|---|---|
| 網羅 | 全受け入れ基準に対応するテストケースがある |
| 異常系 | 異常系・境界値テストが正常系と同数以上ある |
| 再現性 | バグは再現手順・期待値・実際・環境のセットで記録されている |
| 自動化 | 回帰スイートがCIで常時実行され、フレーキーテストが放置されていない |
| 独立性 | テストが実行順序に依存しない |

### 8. Security / Privacy品質

| 基準 | 判定文 |
|---|---|
| 脆弱性 | OWASP Top 10 検査済み・Critical / High がゼロ |
| 認証認可 | 全エンドポイント・全画面で権限昇格・未認可アクセスができない |
| 秘密管理 | APIキー・認証情報がリポジトリ・クライアントコードに含まれない |
| データ最小化 | 収集する個人情報が目的に必要な最小限である |
| ポリシー整合 | 実装とプライバシーポリシー・利用規約が一致している |
| 依存関係 | 既知の重大脆弱性を持つ依存パッケージがない |

### 9. Performance品質

| 基準 | 判定文 |
|---|---|
| 実測主義 | 計測は実機・実回線条件で行われている（ローカル値で判定しない） |
| 目標達成 | Core Web Vitals・API応答が数値基準（下表）を満たす |
| 負荷耐性 | 想定ピークの2倍の負荷で応答時間・エラー率が基準内 |
| 検証記録 | 改善ごとに前後の計測値が記録されている |

### 10. Analytics品質

| 基準 | 判定文 |
|---|---|
| 計測正確性 | 計測値の正しさが検証済み（テスト計測との一致確認） |
| 定義明文化 | 全KPIに定義・集計ロジック・目標値が文書化されている |
| 示唆変換 | レポートが数値の羅列でなく「So What（次の問い・改善候補）」を含む |
| 本質指標 | バニティメトリクスでなくNorth Star Metricに接続する指標を追っている |

| カテゴリ | 指標 | 基準値 |
|---|---|---|
| **Performance** | LCP（Largest Contentful Paint） | ≦ 2.5秒 |
| | INP（Interaction to Next Paint） | ≦ 200ms |
| | CLS（Cumulative Layout Shift） | ≦ 0.1 |
| | API応答時間（主要エンドポイント p95） | ≦ 500ms |
| | 負荷テスト | 想定ピーク×2で エラー率 < 1% |
| **Accessibility** | コントラスト比（通常テキスト） | ≧ 4.5 : 1 |
| | コントラスト比（大テキスト） | ≧ 3 : 1 |
| | タッチターゲット | iOS ≧ 44pt / Android・Web ≧ 48dp |

| | コンソールエラー（本番ビルド） | 0件 |
| | テストカバレッジ | プロジェクト憲章で定義（推奨: 主要ロジック ≧ 80%） |
| **Security** | Critical / High 脆弱性 | 0件 |
| **Test** | 受け入れ基準のテスト網羅率 | 100% |
| | Critical / High バグ残存 | 0件 |

## 品質判定システム

すべての成果物は **PASS / WARNING / FAIL** で判定する（詳細運用: [`Agent_Architecture.md — Quality Control`](./Agent_Architecture.md#quality-control)）。

| 判定 | 本書における基準 |
|---|---|
| ✅ **PASS** | 該当する領域別基準＋数値基準をすべて満たし、証拠が添付されている |
| ⚠️ **WARNING** | 必須基準は満たすが、軽微な懸念・改善余地がある（Open Issuesに記録し放置禁止・持ち越しは2Phaseまで） |
| ❌ **FAIL** | 必須基準の未達が1つでもある。差し戻して修正後に再判定 |

**判定の鉄則**:
1. 判定者は作成者以外（自己承認禁止）
2. 証拠必須（チェックリスト・計測結果・テスト結果）
3. FAILの数値基準を「今回だけ例外」で通さない — 基準を変えるなら正式にPRで改訂する

## ゲート基準

Workflowの3つのゲート（Phase 06 / 13 / 15）は、本書の基準を以下の組み合わせで判定する。

| ゲート | 適用する領域別基準 | 通過条件 |
|---|---|---|
| **Phase 06: Design Review** | 共通5基準＋ UX / UI / Accessibility（設計上の担保） | 全項目PASS＋ユーザビリティテスト重大問題ゼロ＋Human承認 |
| **Phase 13: QA Review** | 共通5基準＋ UX / UI / Accessibility / Code / AI / Test / Performance / SEO / Analytics / Security（一次） | 10領域すべて合格基準達成＋Human承認 |
| **Phase 15: Security Review** | Security / Privacy ＋ AI安全性 | Critical / High ゼロ＋ポリシー整合＋Human承認（リスク受容含む） |

## 品質KPIと継続改善

### 品質を測るKPI

| KPI | 定義 | 目標 |
|---|---|---|
| 初回PASS率 | 初回提出でPASSした成果物の割合 | ≧ 80% |
| ゲート差し戻し回数 | ゲート（06/13/15）でのFAIL回数 / プロジェクト | ≦ 2回 |
| WARNING持ち越し違反 | 2Phaseを超えて持ち越されたWARNINGの件数 | 0件 |
| リリース後Critical障害 | リリース後30日以内のCritical障害件数 | 0件 |
| 基準改訂の還元 | 障害・差し戻しの再発防止が本書に反映された件数 | 全Critical事象で100% |

### 継続改善のルール

1. **失敗は基準に還元する** — リリース後の障害・ゲート差し戻しが起きたら、「どの基準があれば防げたか」を分析し、本書または該当チェックリストに追記する。
2. **形骸化した基準は削除する** — 誰も参照しない・常にN/Aになる基準は四半期レビューで削除する。基準は少なく・鋭く保つ。
3. **改訂はPRで** — 本書の変更はすべてPull Request＋Owner承認で行い、変更理由をChangelogに残す。
